## 1 目标、约束与非目标

### 1.1 架构目标

本设计为 V1-A（0–6 周）提供一个可快速交付、可审计、可降级、可演进的技术基线，支撑：个人账户和唯一工作空间、项目/任务/客户管理、AI COO 任务拆解与每日三件事、独立授权、数据导出/注销、操作审计与基础运营观测。

架构必须满足以下原则：

1. **模块化单体优先**：团队预计不超过 10 FTE、并发目标 1k，先保持单一可部署业务应用与清晰模块边界；不预建微服务、事件总线或插件运行时。
2. **业务写入优先于 AI**：所有业务对象由确定性服务维护；AI 只生成建议，必须经用户确认后才由业务服务写入。
3. **工作空间隔离强制执行**：所有业务资源均带 `workspace_id`，服务层和数据访问层均校验归属；客户端过滤不构成安全边界。
4. **审计与授权内建**：审计、AI traceId、授权变更和注销流程不是后补功能，而是核心请求链路的一部分。
5. **故障不阻断经营**：模型或第三方依赖不可用时，用户仍可手动管理项目、任务和客户。

### 1.2 V1-A 强约束

| 约束 | 架构响应 |
|---|---|
| PWA、移动浏览器优先 | Web 前端采用响应式设计与 PWA 能力；不建设 Native App。 |
| 中国大陆数据存储 | 生产数据库、对象存储、备份、日志与模型接入均需满足地域和合同约束。 |
| 商业 LLM API | 通过供应商适配层调用；敏感字段最小化、脱敏；供应商合同禁止默认训练/留存用户数据。 |
| 1k 并发用户 | 无状态应用横向扩展；数据库连接池、缓存、限流与异步作业分离。 |
| 核心 P95 性能 | 查询 <=500ms、普通写 <=300ms、AI 首 token <=3s、任务生成 <=15s。 |
| 数据权利 | 授权独立留痕；基础导出；注销 30 天宽限与后续匿名化/删除。 |

### 1.3 非目标

本设计不实现或不预建 V1-B/V2/V3 能力：协作市场、站内沟通、支付/结算、内容发布与轻量审核实验、AI CLO、长期记忆、RAG、通用文件服务、企业多成员、插件 SDK/沙箱、搜索中心、工作流引擎、服务网格及微服务拆分。

## 2 总体架构

### 2.1 逻辑视图

```text
┌───────────────────────────────────────────────────────────────┐
│ Web PWA（用户端 / 运营后台）                                   │
│ 注册、工作台、项目、任务、客户、设置、聚合指标                 │
└──────────────────────────┬────────────────────────────────────┘
                           │ HTTPS / JSON / 流式 AI 响应
┌──────────────────────────▼────────────────────────────────────┐
│ Edge / 网关层                                                   │
│ TLS、HSTS、WAF、IP/用户限流、请求大小限制、requestId            │
└──────────────────────────┬────────────────────────────────────┘
                           │
┌──────────────────────────▼────────────────────────────────────┐
│ V1-A 模块化业务应用（无状态）                                  │
│ Auth │ Workspace │ Profile │ Project/Task │ CRM │ AI COO        │
│ Consent/Export/Deactivation │ Audit │ Operations │ Notification*│
│ *V1-A 不提供站内通知，仅保留应用内提示接口                     │
└───────┬──────────────┬────────────────┬────────────────────────┘
        │              │                │
┌───────▼──────┐ ┌─────▼─────┐ ┌────────▼─────────┐
│ PostgreSQL   │ │ Redis     │ │ Worker            │
│ 业务/授权/   │ │ 限流、    │ │ 导出、AI 重试、   │
│ 审计元数据   │ │ 会话、队列│ │ 清理/注销作业     │
└──────────────┘ └───────────┘ └───────┬───────────┘
                                       │
                       ┌───────────────▼───────────────┐
                       │ 外部受控依赖                    │
                       │ LLM API、可选实名服务、邮件服务 │
                       └────────────────────────────────┘
```

### 2.2 关键请求链路

**普通业务写入**：用户请求 → 身份认证 → 资源授权 → 输入校验 → 业务事务 → 审计追加写入 → 返回结果。  
**AI 建议**：用户触发 → 鉴权/授权校验 → 受控读取业务上下文 → 脱敏与 Prompt 组装 → 模型调用 → 结构校验 → 保存建议/trace → 用户确认 → 业务事务写入 + 审计。  
**异步作业**：业务请求仅创建可追踪作业记录；Worker 幂等执行、最多 3 次重试，最终失败进入失败队列并生成可运营告警。

### 2.3 分层规则

| 层 | 职责 | 禁止事项 |
|---|---|---|
| 表现层 | 页面状态、表单交互、调用 API、展示 AI 建议 | 不做权限判断或业务状态裁决 |
| API/控制器 | 身份解析、参数校验、请求/响应映射 | 不直接查询数据库或编排复杂业务 |
| 应用服务 | 用例编排、事务边界、状态机、授权调用 | 不依赖 HTTP 对象或前端类型 |
| 领域模块 | 项目、任务、客户、授权、审计等规则 | 不调用外部 LLM 或直接写日志系统 |
| 仓储/适配器 | 数据访问、缓存、队列、LLM/实名等外部调用 | 不包含业务决策 |

## 3 模块边界与职责

| 模块 | 主要职责 | 核心对象 | V1-A 公开能力 |
|---|---|---|---|
| Auth | 注册、登录、密码重置、会话签发/刷新/撤销、重认证 | User、Session、Credential | 账户认证与安全设置 |
| Workspace/Profile | 唯一工作空间、画像、阶段标签、身份公开范围 | Workspace、Profile | 创建/更新/查看身份 |
| Project/Task | 项目、三级个人任务、状态机、进度计算 | Project、Task | 手动 CRUD、状态迁移 |
| CRM | 客户与阶段、跟进记录 | Customer、CustomerStage | 客户 CRUD、阶段统计 |
| AI COO | 任务拆解、每日三件事、建议确认、模型适配与降级 | AiRun、AiSuggestion | 请求建议、确认/驳回建议 |
| Consent & Lifecycle | 授权、撤回、导出、注销/恢复 | Consent、ExportJob、Deactivation | 权限管理、导出、注销 |
| Audit | 业务操作审计、AI 追踪关联、不可变追加记录 | AuditEvent、Trace | 内部写入、受控运营查询 |
| Operations | 用户状态、受控用户查询、聚合指标、AI 配额 | MetricAggregate、QuotaPolicy | 运营后台；当前 V1-A 基线不包含内容审核 |

模块仅可经服务接口依赖其他模块；禁止跨模块直接读写对方的表。V1-A 可共享同一 PostgreSQL 实例，但每张业务表必须标明所属模块和数据归属字段。

## 4 推荐技术基线与 ADR

### 4.1 默认实施基线

以下为满足 V1-A 的推荐默认实施基线，均应在项目启动会确认并固化为 ADR；替换实现不得削弱本文件的安全、审计和性能约束。

| 层面 | 推荐基线 | 选择原因 |
|---|---|---|
| 代码仓库 | TypeScript Monorepo | 前后端共享领域类型和校验规则，降低小团队协作成本。 |
| 前端 | React Web PWA（Next.js 或等价 SSR/CSR 框架） | 支持移动 Web、首屏性能和 PWA 发布。 |
| 后端 | TypeScript 模块化 REST 服务（Fastify/NestJS 或等价框架） | 适合模块化单体、输入校验、可观测中间件与流式响应。 |
| 关系数据 | PostgreSQL | 事务、行级数据约束、JSON 扩展、审计关联和导出适用。 |
| 缓存/队列 | Redis + 受控队列 Worker | 用于限流、短期会话、异步作业和失败重试。 |
| AI | 模型供应商适配器 + 严格结构化输出校验 | 避免供应商锁定，保证任务建议可校验。 |
| 可观测性 | OpenTelemetry 兼容 trace、结构化日志、指标系统 | 将 HTTP、异步作业和 AI run 以 traceId 关联。 |
| 部署 | 容器化、托管运行平台、托管 PostgreSQL/Redis | 将运维复杂度控制在 V1-A 可承担范围。 |

> 本文不锁定具体云厂商、模型供应商、框架小版本或 ORM；这些应在采购、地域合规、团队熟练度和成本评估后通过 ADR 确认。

### 4.2 ADR 清单

| 编号 | 决策 | 建议结论 | 状态 |
|---|---|---|---|
| ADR-01 | 应用拆分方式 | V1-A 采用模块化单体 + 独立 Worker，不采用微服务。 | 建议确认 |
| ADR-02 | 会话方式 | 短时 Access Token + Refresh Token 轮换；Refresh Token 仅置于 HttpOnly、Secure、SameSite Cookie，并服务端可撤销。 | 建议确认 |
| ADR-03 | 数据隔离 | 应用服务强制 `workspace_id` 过滤；数据库外键/复合唯一约束辅助；高风险查询加入数据库行级策略作为纵深防御。 | 建议确认 |
| ADR-04 | AI 模型接入 | 至少支持两家供应商的适配接口；生产只启用已完成数据合同审查的供应商。 | 待采购确认 |
| ADR-05 | AI 编排 | V1-A 使用单 Agent、受控业务查询和显式确认状态机；不引入通用多 Agent 框架。 | 建议确认 |
| ADR-06 | 异步任务 | 导出、AI 后台重试、注销到期处理使用队列 Worker；作业幂等、最多重试 3 次。 | 建议确认 |
| ADR-07 | 审计存储 | 业务审计使用 append-only 审计表；数据库写权限限制为追加；后续规模化后再评估外部不可变存储。 | 建议确认 |
| ADR-08 | 实名服务 | V1-A 基线不启用，仅预留适配接口；接入前完成供应商与数据处理评估并修订相关文档。 | 已决策；后续启用需重新评审 |
| ADR-09 | 运营身份与数据留存 | 独立 `platform_operator` + 最小权限/MFA；审计 13 个月、应用日志 180 天、AI 脱敏摘要 90 天、备份 30 天自然覆盖、导出 7 天。 | 已决策；生产前法务书面确认留存期限 |

## 5 身份认证、会话与权限

### 5.1 认证流程

1. 用户以手机或邮箱注册；密码使用 Argon2id 或 bcrypt 哈希保存，绝不保存明文。
2. 登录成功后签发短时 Access Token，并签发带唯一会话标识的 Refresh Token。
3. Refresh Token 仅通过 `HttpOnly + Secure + SameSite` Cookie 传输；服务端保存其哈希、设备摘要、签发/到期/撤销状态与轮换链。
4. 刷新时执行轮换：旧令牌立即失效，新令牌绑定新会话版本；检测到已使用的失效刷新令牌时撤销该会话族并要求重新登录。
5. 修改密码、申请注销等重要操作要求最近认证；失败登录达到 5 次后锁定 15 分钟。

### 5.2 授权模型

V1-A 采用“账户角色 + 资源归属”的 RBAC/ABAC 混合模型：

- 普通用户只访问 `workspace.owner_user_id = current_user_id` 的工作空间；
- 项目、任务、客户、导出和授权记录均从工作空间派生归属；
- 运营角色仅访问其被授予的后台资源，且默认无权读取用户私密业务正文；
- 每个资源查询和写入均在服务层显式传入当前主体与工作空间上下文；禁止由客户端传入的 `workspace_id` 直接决定访问边界。

### 5.3 安全中间件顺序

`requestId → 结构化日志 → TLS/安全头 → CORS → IP/用户限流 → 请求体限制 → 身份认证 → 资源授权 → 输入校验 → 业务处理 → 审计 → 统一错误响应`。

错误响应使用稳定错误码与用户可读中文提示，不返回堆栈、SQL、模型 Prompt 或内部拓扑信息。

## 6 数据架构与隔离

### 6.1 核心数据域

| 数据域 | 关键实体 | 关键约束 |
|---|---|---|
| 身份 | User、Credential、Session | 用户唯一标识；Token 仅存哈希；会话可撤销。 |
| 工作空间与画像 | Workspace、Profile、VisibilitySetting | 一用户一工作空间；工作空间名称改名次数受产品规则约束。 |
| 经营 | Project、Task、Customer、CustomerStageHistory | 每条记录有 `workspace_id`、创建/更新时间和版本字段。 |
| AI | AiRun、AiSuggestion、AiSuggestionDecision | 每次调用有 traceId、模型版本、输入/输出摘要、授权依据。 |
| 治理 | Consent、ExportJob、DeactivationRequest | 授权按类型独立记录，支持撤回和生命周期状态。 |
| 审计 | AuditEvent | 只追加；记录操作者、动作、对象、前后状态摘要、traceId。 |

### 6.2 多层隔离策略

1. **入口层**：从认证上下文确定 `user_id`，不信任客户端的主体信息。
2. **服务层**：所有资源操作以 `workspace_id + actor_id` 组合验证归属和权限。
3. **查询层**：所有业务表查询默认注入 `workspace_id` 条件；禁止提供裸主键查询仓储接口。
4. **数据库层**：外键引用工作空间；关键表采用复合唯一索引；生产环境对高风险业务表启用或评估 Row-Level Security 作为防御纵深。
5. **审计与测试**：越权测试覆盖 URL/ID 篡改、批量查询、导出、异步作业和运营后台。

### 6.3 事务与并发

项目状态、任务状态、AI 建议确认及对应审计事件必须在同一数据库事务中提交。业务对象使用乐观锁版本号或等价条件更新，避免重复确认、重复完成或并发覆盖。关键写操作支持幂等键；同一幂等键在有效窗口内必须返回同一业务结果。

### 6.4 导出与注销

- 导出作业只读取当前工作空间的项目、任务、客户基础数据，生成 CSV；下载链接一次性且仅保留 7 天。
- 注销申请立即冻结新业务写入与新 AI 调用，但允许用户在 30 天内查看历史数据和撤销申请。
- 宽限期届满由 Worker 执行删除/匿名化策略，并记录保留例外、依据与处理结果；备份数据在其保留周期内覆盖清除。
- 存在争议、投诉或法定留存义务时，系统保存最小必要数据并对用户显示原因和预计期限。

## 7 AI COO 架构

### 7.1 V1-A 能力边界

AI COO 仅支持：项目任务拆解、基于当前结构化项目/任务/客户数据的每日三件事建议，以及低置信度时的澄清问题。它使用单次会话上下文，不使用跨会话长期记忆、向量检索或 RAG。

### 7.2 受控调用流程

```text
用户触发
  → 认证与 AI COO 数据授权校验
  → 读取当前 workspace 的最小必要结构化数据
  → 识别/脱敏敏感字段，拒绝不可安全发送的数据
  → 系统指令 + 结构化上下文 + 用户输入分区组装
  → 模型适配器调用（限额、超时、重试）
  → 严格 JSON Schema 校验 / 低置信度澄清
  → 写入 AiRun、AiSuggestion、traceId 和审计事件
  → 用户查看、编辑、确认或驳回
  → 确认后由 Project/Task 服务事务化写入
```

### 7.3 工具和数据访问控制

AI 不直接获得数据库连接、HTTP 管理令牌或通用代码执行能力。AI 只能通过经审核的只读工具读取当前工作空间的结构化项目、任务、客户摘要；工具参数由服务端从认证上下文派生工作空间，模型不得自行指定其他用户或工作空间标识。

V1-A 不开放“自动创建/修改/删除”工具。AI 的任何建议均以草稿或待确认对象保存；确认动作由用户 UI 发起并触发确定性业务服务。

### 7.4 结构化输出与失败处理

| 场景 | 处理 |
|---|---|
| 首 token 超过目标或总时长超过 15 秒 | 返回处理中提示，后台最多重试一次；用户可手动管理任务。 |
| 输出不符合任务 Schema | 自动修复/重试一次；仍失败则降级为明确标记的自然语言建议，不自动创建任务。 |
| 关键输入不足或置信度低 | 返回澄清问题，不编造事实。 |
| 模型供应商失败 | 熔断/降级，保留错误类别与 traceId，不泄露供应商内部错误。 |
| 配额耗尽 | 明示调用限额已达上限；手动经营功能保持可用。 |

### 7.5 AI 安全与审计

- 外部输入、检索/业务数据与系统指令采用结构化消息分区，禁止字符串拼接为系统指令。
- 对输入实施大小限制、敏感信息识别、Prompt 注入规则检测与工具参数白名单校验。
- 每个 AI run 记录 traceId、调用时间、用户/工作空间、模型/提示词版本、输入输出摘要、token 消耗、数据类型/授权依据、结果状态和用户决策。
- 原始 Prompt、原始输出及敏感字段的留存、脱敏和访问权限应按数据治理策略控制；日志不得输出密码、令牌、原始实名证件或未脱敏客户信息。

## 8 审计、可观测性与运营数据

### 8.1 统一关联标识

每个 HTTP 请求创建 `requestId` 与可传播 `traceId`。用户操作、业务事务、异步作业、AI 调用与错误日志均携带 traceId；异步作业生成新的 span 但保留父 traceId。

### 8.2 审计事件最小字段

| 字段 | 说明 |
|---|---|
| event_id / occurred_at | 不可变事件标识与发生时间 |
| actor_type / actor_id | 用户、管理员、Worker 或系统 |
| workspace_id | 数据归属范围（如适用） |
| action / resource_type / resource_id | 操作及对象 |
| before_summary / after_summary | 脱敏后的状态摘要，非完整私密正文 |
| trace_id / request_id | 关联请求和 AI 链路 |
| result / failure_code | 成功、拒绝、失败及稳定错误码 |

审计必须覆盖注册/登录/注销、项目和任务写操作、授权变更、数据导出、管理员操作、AI 建议生成和确认/驳回。V1-A 先保证记录可用，完整查询 UI 推迟至 V1-B。

### 8.3 指标与告警

| 维度 | 指标示例 |
|---|---|
| 产品阶段门 | 注册→项目创建、AI 确认率、7 日留存、每日三件事周活、任务完成率 |
| AI 质量 | 结构化输出成功率、Schema 校验失败率、超时率、用户编辑/驳回率、单用户成本 |
| 平台健康 | 请求量、P95/P99 延迟、错误率、数据库连接、队列堆积、缓存命中率 |
| 安全 | 登录失败锁定、限流命中、鉴权拒绝、越权测试结果、Prompt 注入拦截 |
| 作业 | 重试次数、死信数量、导出成功率、注销到期处理积压 |

告警应至少覆盖：核心接口错误率、AI 超时/失败异常上升、队列积压、数据库不可用、备份失败和高危安全事件。

## 9 可靠性、容量与灾备

### 9.1 容量与伸缩

业务应用保持无状态，通过水平扩容应对 1k 并发。数据库连接池设置上限和超时；AI 调用设置用户级、能力级及全局并发上限，避免单个用户或供应商波动耗尽资源。长耗时工作（导出、重试、注销处理）只能在 Worker 执行。

### 9.2 可靠性策略

| 风险 | 策略 |
|---|---|
| LLM 不可用 | 超时、熔断、一次异步重试、手动任务管理降级；不阻塞项目/CRM。 |
| Redis/队列故障 | 不将 Redis 作为业务事实来源；同步业务写入继续使用数据库；异步能力明确提示延迟。 |
| 数据库故障 | 健康检查、连接超时、托管高可用与恢复演练。 |
| 实名等第三方故障 | V1-A 非核心流程，显示明确失败并允许稍后重试。 |
| 重复消息 | 消费端按作业 ID/业务幂等键实现幂等。 |

### 9.3 备份与恢复

核心数据每日备份，至少保留 30 天；应用日志默认保留 180 天；审计事件热数据保留 13 个月；AI 脱敏调用摘要默认保留 90 天；原始 Prompt/输出不进入常规留存。RPO 不超过 24 小时，RTO 不超过 4 小时。备份恢复、导出安全和注销清理需在发布前演练。恢复演练不得将生产用户数据复制到无等价安全控制的环境。

### 13.2 认证与会话补充（落实 ADR-02 条件）

1. **SameSite 取值**：Refresh Token Cookie 固定 `SameSite=Lax`（兼容顶层导航进入 PWA 既有会话，同时阻跨站 POST）。不允许 `SameSite=None`。如未来出现第三方嵌入场景再评估升级为 `Strict`。
2. **CSRF 防护**（响应 SRS `NFR-SEC-15` 与 PRD 安全基线）：
   - 所有写操作 API（POST/PUT/PATCH/DELETE）必须执行 **`Origin` 头校验**，不在白名单的请求一律 403。
   - 所有写操作 API 必须要求自定义请求头 `X-OPC-CSRF: 1`，由前端在每次写请求时携带，浏览器跨站简单表单无法伪造该头。
   - CORS 仅允许 PWA 同源与受控运营后台域名；禁止 `Access-Control-Allow-Origin: *`。
   - §5.3 中间件链更正为：`requestId → 结构化日志 → TLS/安全头 → CORS → Origin/CSRF 校验 → IP/用户限流 → 请求体限制 → 身份认证 → 资源授权 → 输入校验 → 业务处理 → 审计 → 统一错误响应`。
3. **会话族结构**：`Session` 表（在 04 数据模型中定义）必须包含 `session_family_id` 与 `session_version`，由同一 Refresh Token 链派生的会话共享 family；发现重放失效令牌时撤销该 family，该会话族对应的设备需重新登录，不误撤销其他独立设备的 family。
4. **重认证窗口**：修改密码、申请注销、撤销授权等敏感操作的"最近认证"窗口定义为 **15 分钟**，并绑定**当前 Session**，不得存为用户级共享时间戳。超时须在当前会话重新走密码或 OTP 验证后才能继续；一个设备的重认证不能授权其他设备执行敏感操作。
5. **登录失败锁定维度**：按 **账号 + IP** 双维度计数，单维度达 5 次锁定该维度 15 分钟；两者均命中则触发账号整体冻结并通知用户。防被恶意锁定他人账号：账号锁定可通过注册邮箱/手机验证解锁，不依赖服务端客服。

### 13.3 审计落地补充（落实 ADR-07 条件）

1. **Append-only 强制**：`audit_event` 表通过 PostgreSQL 触发器拒绝 `UPDATE` / `DELETE`（V1-A 即启用，而非"后续评估"）。
2. **角色与事务写入**：`audit_event` 表的直接 `INSERT/UPDATE/DELETE` 权限仅授予表所有者 `audit_owner`，业务应用和 Worker 角色均无直接表写权限。数据库提供受控 `SECURITY DEFINER` 函数 `append_audit_event(...)`（函数所有者为 `audit_owner`，固定 `search_path`，校验摘要白名单与调用参数），仅向业务应用/Worker 授予 `EXECUTE`。业务服务在原业务事务的同一连接中调用该函数，确保业务写与审计原子提交；不得使用独立连接或事务外审计写入。
3. **分区策略**：`audit_event` 按月分区，至少保留 13 个月热数据；冷分区迁移至独立归档存储（V1-A 可暂用同实例独立表空间，V1-B 评估外部不可变存储）。
4. **审计字段白名单**（落实 PRD 埋点字典约束）：审计 `before_summary` / `after_summary` 仅允许写入以下字段类型——标识符、枚举状态、时间戳、计数、布尔。**禁止**写入：密码、令牌、原始身份证件、客户完整备注、AI 原始 Prompt/原始输出、未脱敏客户信息。任务标题、客户名称进入审计时按字段长度截断至 64 字符并标注 `truncated` 位。
5. **AI traceId 关联**：`audit_event` 中涉及 AI 的条目必须携带 `ai_run_id`（外键 `AiRun.id`）与 `trace_id`，运营查询可通过该外键跳转至 AI 调用明细。

### 13.4 AI 调用与降级补充

1. **删除"自动修复"措辞**：§7.4 "自动修复/重试一次"更正为 **"最多重试 1 次相同输入；若仍失败则降级为标记自然语言建议，绝不自动创建业务对象"**。不存在"自动修复"路径。
2. **AI 限流基线**（§9.1 落地值）：
   - 用户级：单用户 5 QPS、单日 200 次调用（V1-A 冷启动期可调，调整须由运营后台变更并留痕审计）。
   - 能力级：每日三件事单用户每日最多 3 次；任务拆解单项目每日最多 10 次。
   - 全局：50 QPS、当日累计 5000 次。达全局上限时返回稳定错误码 `AI_QUOTA_GLOBAL_EXHAUSTED`，不阻塞手动业务。
3. **脱敏规则交接**：§7.2 "脱敏敏感字段"的具体算法（删除 / Token 化 / 长度截断）在 **04 数据模型** 中以字段级表格定义，本架构文档不重复。
4. **模型供应商故障熔断**：连续 5 分钟内失败率 ≥30% 触发熔断 60 秒，期间所有 AI 调用直接走"手动管理任务降级"路径，不阻塞项目/CRM/客户写入。熔断恢复后第一次调用作为探针，失败则再次进入熔断。

### 13.5 注销与备份窗口协调补充（落地 §6.4 + §9.3）

1. **注销主库状态机**（04 数据模型中细化状态枚举）：
   - `ACTIVE`（正常）→ `REQUESTED`（提交注销，冻结新写入/AI 调用）
   - → `GRACE`（30 天宽限期，可恢复，可登录只读）
   - → `TOMBSTONING` → `TOMBSTONED`（宽限期满 Worker 匿名化主库字段，保留 `user_id` + 注销原因 + 法定留存依据）
   - `GRACE` → `REVOKED`（宽限期内撤销，账户回到 `ACTIVE`，留审计）
2. **主库 vs 备份解耦**：
   - 主库：宽限期满后写 tombstone（字段匿名化），不再可恢复。
   - 备份：沿用自然保留周期（30 天），**不主动清除**；但在每次新备份生成时，已 tombstone 的用户记录直接以匿名形态写入新备份，原备份期满自然覆盖。
   - 恢复演练：仅恢复到无 tombstone 的快照副本演练环境，禁止将含 tombstone 的备份恢复到与生产有等价访问控制的环境。
3. **法定留存例外**：存在争议/投诉/法定留存时，主库写 `retention_hold` 字段（在 04 数据模型 `deactivation_request` 表上），`GRACE` 转为 `RETENTION_HOLD`，不自动进入 `TOMBSTONING`；向用户在 UI 显示原因与预计期限，留审计留痕。

### 13.6 数据隔离范围下调说明（明确 ADR-03）

- V1-A 仅使用应用层强制 `workspace_id` 过滤 + DB 复合唯一索引 + 服务层组合校验；
- **PostgreSQL RLS 在 V1-A 不启用**；理由：1k 并发单租户工作空间模型下，应用层强制已足够，RLS 增调试复杂度与误配置风险，且阶段门无对应验收能力。
- V1-B 进入多成员/多角色权限前，重新评估 RLS 作为纵深防御。

### 13.7 可观测性范围下调说明（明确 §8.1）

- V1-A **不部署完整 OpenTelemetry collector pipeline**；
- V1-A 关联机制：HTTP 请求头/日志携带 `requestId` + `traceId`；AI 调用与异步作业在数据库行上持久化 `trace_id` 字段；结构化日志（JSON）写入时附带这两个标识；
- 完整分布式 OTel 与后端（Tempo/Jaeger）作为 V1-B 评估项，本架构不强制在 V1-A 实施。

### 13.8 指标与性能口径下沉

- **§8.3 阶段门指标**：完整分母/分子定义见 SRS §6.7 与 PRD §1.6，本架构不再重复；
- **性能 P95 测量口径**：
  - 普通 API P95：从 Edge 网关收到请求到响应返回，<500ms。
  - AI 首 token ≤3s：从用户触发 AI 请求到第一批流式响应可消费。
  - AI 任务生成 ≤15s：从触发到 `AiRun` 终态（含 Schema 校验失败降级）。
  - 测量工具与采样位置在 **07 测试与发布计划** 中定义，本架构不强制工具选型。
