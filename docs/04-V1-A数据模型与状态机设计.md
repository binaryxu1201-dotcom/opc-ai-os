## 1 目标、范围与设计原则

### 1.1 目标

本设计将 V1-A 已冻结的业务规则落为 PostgreSQL 可实现的数据模型、约束、索引、迁移顺序及状态机。它是后续 `05-V1-AAPI契约设计.md`、应用服务和数据库迁移的唯一数据契约来源。

### 1.2 V1-A 范围

本文件覆盖：账户/凭据/会话、独立平台运营身份、唯一工作空间与画像、项目/三级个人任务、客户及阶段历史、AI 运行与建议、授权、导出、注销、异步作业、审计、幂等与聚合运营数据。

以下对象不建表、不预留业务流程：多成员协作、需求发布、合作/结算/评价、支付、站内沟通、RAG/向量库/长期记忆、通用文件、企业主体、内容社区实验。V1-B 兼容仅限 `task.assignee_user_id`、可扩展 JSON 字段及稳定的外键边界，不提前创建 V1-B 表。

### 1.3 强制原则

1. **工作空间归属**：所有用户私有业务表必须有 `workspace_id`；仓储禁止提供只按裸 `id` 查询业务资源的方法。
2. **应用层隔离**：V1-A 强制服务层传入 `actor_id + workspace_id` 并验证归属；不启用 PostgreSQL RLS（架构 §13.6）。
3. **确定性写入**：AI 运行和建议不直接修改 `project`、`task`、`customer`。只有用户确认后，业务服务在同一事务中写业务对象、建议决议和审计事件。
4. **不可变留痕**：`audit_event` 只追加；关键状态变化和授权变更必须有审计记录。
5. **并发安全**：所有业务可写表使用 `version` 乐观锁；关键写请求使用 `idempotency_key`。
6. **最小化与可删除**：不保存原始身份证件；AI 原始输入/输出不进常规业务表；注销主库匿名化与备份自然覆盖按 §10 执行。

## 2 通用约定

### 2.1 数据库、命名与通用字段

| 项目 | 约定 |
|---|---|
| 数据库 | PostgreSQL 16+；默认 schema 为 `public`，逻辑模块以表前缀/迁移目录区分。 |
| 身份表命名 | 领域实体称 `User`，PostgreSQL 物理表固定为 `app_user`，避免使用 `user` 关键字/当前用户语义；下文 SQL 外键均引用 `app_user(id)`。 |
| 主键 | `uuid`，应用端使用 UUIDv7 生成；不得使用可枚举自增主键作为外部资源标识。 |
| 时间 | `timestamptz`，统一 UTC；API 在展示层转用户时区。 |
| 枚举 | PostgreSQL `text` + `CHECK`，便于 expand/contract；值为大写 snake case。 |
| 业务可写表 | `id`、`workspace_id`（适用时）、`created_at`、`updated_at`、`created_by_user_id`、`updated_by_user_id`、`version`。 |
| 软删除 | V1-A 核心经营数据不使用通用 `deleted_at`；用户可取消项目/任务，账户删除走注销状态机。 |
| 扩展字段 | 仅 `metadata jsonb NOT NULL DEFAULT '{}'`；不得把核心筛选、隔离、状态或授权字段放进 JSON。 |
| 文本 | 用户可编辑文本均在 API 层限制长度；数据库用 `text` + `CHECK (char_length(...) <= n)` 兜底。 |

`version` 初始为 `1`；更新 SQL 必须含 `WHERE id = :id AND workspace_id = :workspaceId AND version = :expectedVersion`，成功后 `version = version + 1`。零行更新返回稳定并发冲突错误，不静默覆盖。

### 2.2 归属、外键与复合约束规则

1. `workspace` 由单一 `owner_user_id` 归属，`owner_user_id` 唯一，确保“一账户一工作空间”。
2. 子资源外键引用父资源时，同时保留 `workspace_id`；数据库通过 `(id, workspace_id)` 复合唯一键和复合外键阻断跨工作空间引用。
3. `user_id` 是身份主体，`workspace_id` 是业务数据边界；运营账户不能以管理员身份绕过业务表查询，运营统计只读聚合表。
4. 业务表的 `created_by_user_id` / `updated_by_user_id` 默认引用 `app_user(id)`；注销后该用户行匿名化但主键保留，保证审计和外键可追溯。
5. `ON DELETE CASCADE` 仅允许用于从属且无独立留存义务的数据（如 `profile`、`customer_stage_history`）。业务数据、审计、AI、导出、注销记录不得通过级联删除消失。

### 2.3 通用审计与幂等规则

| 场景 | 规则 |
|---|---|
| 关键写入 | 项目/任务/客户写、AI 确认、授权变更、导出、注销必须在同一事务追加 `audit_event`。 |
| 幂等窗口 | `idempotency_record` 以 `actor_user_id + scope + key` 唯一，TTL 24 小时；同键同请求返回原响应，不同请求摘要返回冲突。 |
| 请求关联 | 每个写入携带 `request_id`；涉及 AI/异步作业另携带 `trace_id`。 |
| 批量更新 | 按行验证并返回逐项结果；单行并发冲突不得回滚已成功的其他行。 |

## 4 实体与字段目录

### 4.1 身份、认证与会话

#### `app_user`（Auth，领域实体 User）

| 字段 | 类型/约束 | 说明 |
|---|---|---|
| `id` | `uuid PK` | 身份主体。 |
| `email` | `citext NULL UNIQUE` | 邮箱注册二选一；需验证后才可登录。 |
| `phone_e164` | `varchar(32) NULL UNIQUE` | 手机注册二选一；按 E.164 规范化。 |
| `status` | `text CHECK` | `ACTIVE`、`DEACTIVATION_REQUESTED`、`DEACTIVATION_GRACE`、`TOMBSTONED`、`LOCKED`。 |
| `display_name` | `varchar(64) NULL` | 注销时置为“已注销用户”。 |
| `email_verified_at` / `phone_verified_at` | `timestamptz NULL` | 至少一种方式验证。 |
| `tombstoned_at` | `timestamptz NULL` | 主库匿名化完成时间。 |
| `created_at` / `updated_at` | `timestamptz NOT NULL` | 通用时间戳。 |
| `version` | `integer NOT NULL DEFAULT 1` | 乐观锁；注销副作用、密码修改、状态变更均以条件更新保护。 |
| `metadata` | `jsonb` | 仅非核心、非敏感扩展。 |

约束：`CHECK (email IS NOT NULL OR phone_e164 IS NOT NULL)`；`status='TOMBSTONED'` 时 `email`、`phone_e164` 置为不可反查的匿名占位值/哈希，不可复用原值。`LOCKED` 状态由登录限流触发，不作为长驻业务状态：详见 §4.1 末尾"登录限流与 LOCKED 状态"。

#### 登录限流与 `LOCKED` 状态

登录失败计数由 **Redis 滑动窗口** 维护（key 形如 `login_fail:{userId}` 与 `login_fail:{ipPrefix}`，TTL 15 分钟），不建 `login_attempt` 长驻表。账号 + IP 双维度达 5 次后：
- 单维度命中 → 仅锁定该维度 15 分钟，Redis TTL 自然到期解除，不写 `user.status='LOCKED'`。
- 两维度同时命中 → 触发账号整体冻结，将 `user.status` 由 `ACTIVE` 临时置为 `LOCKED`，写审计事件（`action=USER_LOCKED`），并允许用户通过注册邮箱/手机验证解锁（解锁后回 `ACTIVE` 并写审计）。
- `LOCKED` 不在注销六态之内，仅作为登录安全中间态；注销流程遇到 `LOCKED` 账户仍允许在重认证后进入 `REQUESTED`，但服务层须先要求解锁。

#### `credential`（Auth）

| 字段 | 类型/约束 | 说明 |
|---|---|---|
| `id` | `uuid PK` | — |
| `user_id` | `uuid FK app_user(id)` | 所属账户。 |
| `credential_type` | `text CHECK` | V1-A 仅 `PASSWORD`；OTP 验证不作为长期凭据保存。 |
| `secret_hash` | `text NOT NULL` | Argon2id/bcrypt 哈希；不得保存明文、可逆密文或重置令牌。 |
| `hash_algorithm` | `varchar(32)` | 便于哈希策略迁移。 |
| `changed_at` | `timestamptz` | 修改密码后撤销会话族。 |
| `status` | `ACTIVE` / `DISABLED` | 不删除历史凭据记录。 |

索引：`UNIQUE (user_id, credential_type) WHERE status = 'ACTIVE'`。

#### `session`（Auth）

| 字段 | 类型/约束 | 说明 |
|---|---|---|
| `id` | `uuid PK` | 会话实例/Refresh Token 标识。 |
| `user_id` | `uuid FK app_user(id)` | 会话主体。 |
| `session_family_id` | `uuid NOT NULL` | 同一 Refresh Token 轮换链共享；重放撤销整族。 |
| `refresh_token_hash` | `text NOT NULL UNIQUE` | 仅存哈希。 |
| `session_version` | `integer NOT NULL` | 每轮换递增。 |
| `parent_session_id` | `uuid NULL FK session(id)` | 前序轮换节点。 |
| `device_summary` / `ip_prefix_hash` | `varchar(256) NULL` | 最小化设备摘要/IP 前缀哈希。 |
| `issued_at` / `expires_at` / `last_used_at` | `timestamptz` | 生命周期。 |
| `last_authenticated_at` | `timestamptz NULL` | 当前 Session 最近重认证时间；敏感操作要求距当前不超过 15 分钟，不得从其他 Session 继承。 |
| `revoked_at` / `revoke_reason` | `timestamptz/text NULL` | `LOGOUT`、`ROTATED`、`REPLAY_DETECTED`、`PASSWORD_CHANGED`、`DEACTIVATED`。 |

索引：`(user_id, revoked_at, expires_at)`、`(session_family_id, revoked_at)`、`(expires_at)`；刷新时以事务锁定当前行，先作废旧会话再创建新会话。轮换产生的新 Session 仅在旧 Session 的 `last_authenticated_at + 15 分钟 > now()` 时复制该时间，否则置空；不得从同用户的其他 Session 或其他 family 读取。

#### `platform_operator` 与 `platform_operator_role`（Operations/Auth）

平台运营身份与普通用户 `app_user`、工作空间归属完全分离，不加入任何 `workspace` 成员关系。运营身份使用独立管理员认证入口；V1-A 最低要求独立账号 + MFA，后续可接企业 IdP。

| 表/字段 | 类型/约束 | 说明 |
|---|---|---|
| `platform_operator.id` | `uuid PK` | 平台运营主体，不是用户业务主体。 |
| `subject_id` | `varchar(128) NOT NULL UNIQUE` | 外部 IdP subject 或受控管理员账号标识；不允许前端自报。 |
| `status` | `ACTIVE` / `SUSPENDED` / `REVOKED` | `REVOKED` 不得登录。 |
| `mfa_enrolled_at` | `timestamptz NOT NULL` | 未完成 MFA 不得激活。 |
| `created_at` / `updated_at` / `version` | 通用字段 | 变更使用乐观锁。 |
| `platform_operator_role` | `operator_id`、`role`、`granted_by`、`granted_at`、`revoked_at`、`version` | 角色：`OPERATIONS_READER`、`PLATFORM_ADMIN`、`SECURITY_AUDITOR`、`SUPPORT`；角色变更必须审计，平台管理员不得自行授予自己更高权限。 |

权限边界：`OPERATIONS_READER` 只能读聚合指标；`PLATFORM_ADMIN` 可配置 AI 配额和查询掩码账户信息；`SECURITY_AUDITOR` 可读必要安全审计摘要但不读业务正文；`SUPPORT` 可查必要非敏感账户信息但不可代用户操作。运营身份不得通过 `workspace_id` 访问用户项目、任务、客户备注或原始 AI 内容。用户搜索至少 4 字符、严格限流、返回掩码联系方式，并为每次查询和配置变更追加管理员审计。

### 4.3 经营域：项目、任务、客户

#### `project`（Project/Task）

| 字段 | 类型/约束 | 说明 |
|---|---|---|
| `id` / `workspace_id` | `uuid PK` / `uuid NOT NULL` | `(id, workspace_id)` 唯一供复合外键引用。 |
| `name` | `varchar(160) NOT NULL` | 项目名称。 |
| `objective` | `varchar(4000) NOT NULL` | 项目目标；主库存储完整 4000 字符，发送给模型时按 §6 截断至 2000。 |
| `deliverable` | `varchar(2000) NULL` | 交付物描述。 |
| `status` | `text CHECK` | `DRAFT`、`IN_PROGRESS`、`PAUSED`、`COMPLETED`、`CANCELLED`；不实现 `ARCHIVED`。 |
| `planned_start_at` / `planned_end_at` | `date NULL` | 计划时间。 |
| `completed_at` / `cancelled_at` | `timestamptz NULL` | 终态时间。 |
| `source` | `MANUAL` / `AI_CONFIRMED` | 创建来源，非状态。 |

约束与索引：`UNIQUE (id, workspace_id)`；`CHECK (planned_end_at IS NULL OR planned_start_at IS NULL OR planned_end_at >= planned_start_at)`；`(workspace_id, status, updated_at DESC)`、`(workspace_id, planned_end_at) WHERE status IN ('DRAFT','IN_PROGRESS','PAUSED')`。

#### `task`（Project/Task）

| 字段 | 类型/约束 | 说明 |
|---|---|---|
| `id` / `workspace_id` / `project_id` | UUID | `project_id + workspace_id` 复合外键引用 `project`。 |
| `parent_task_id` | `uuid NULL` | 同项目、同空间复合外键；最大三级。 |
| `depth` | `smallint NOT NULL CHECK (depth BETWEEN 1 AND 3)` | 根任务为 1，由服务端计算。 |
| `title` / `description` | `varchar(200)` / `varchar(4000) NULL` | 可执行事项。 |
| `assignee_user_id` | `uuid NOT NULL` | 固定 `workspace.owner_user_id`；为 V1-B 保留，不开放成员选择。 |
| `status` | `DRAFT`、`CONFIRMED`、`IN_PROGRESS`、`COMPLETED`、`CLOSED`、`CANCELLED` | 已确认后的业务任务状态。 |
| `due_at` / `estimated_minutes` | `timestamptz NULL` / `integer NULL` | `estimated_minutes > 0`。 |
| `is_deferred` / `is_overdue` | `boolean NOT NULL DEFAULT false` | 附加标记，不替代主状态。 |
| `source` / `source_ai_suggestion_id` / `source_ai_item_key` | `MANUAL` / `AI_CONFIRMED`；UUID NULL；varchar(64) NULL | AI 确认创建时可追溯到建议及其中的稳定条目键。 |
| `completed_at` / `closed_at` / `cancelled_at` | `timestamptz NULL` | 终态时间。 |
| `created_at` / `updated_at` | `timestamptz NOT NULL` | 通用时间戳（§2.1 约定，下文实体表均适用，不再重复列出）。 |
| `version` | `integer NOT NULL DEFAULT 1` | 乐观锁；所有状态转换与字段更新均以条件更新保护。 |

关键完整性：

- `UNIQUE (id, workspace_id)`；`UNIQUE (source_ai_suggestion_id, source_ai_item_key) WHERE source_ai_suggestion_id IS NOT NULL`，允许一个任务计划建议创建多项任务，同时阻止同一建议条目重复写入；AI 来源任务的 `source_ai_item_key` 必填。
- `CHECK (parent_task_id IS NULL OR depth > 1)`；服务层读取父任务并校验 `parent.depth = depth - 1`、父子 `project_id` 相同；数据库触发器再验证，防止绕过 API 写出第 4 层。
- `CHECK (status <> 'CLOSED' OR completed_at IS NOT NULL)`；任务取消不级联取消子任务，服务层须显式要求用户处理非终态子任务。
- 索引：`(workspace_id, project_id, status, due_at)`、`(workspace_id, parent_task_id, created_at)`、`(assignee_user_id, status, due_at)`。

> **AI 草稿边界**：AI 在用户确认前的任务草稿只存于 `ai_suggestion.proposed_payload`，不创建 `task` 行。PRD/SRS 的“待确认/被驳回”逻辑状态由 `ai_suggestion` 承担；确认事务成功后才生成 `task(status='CONFIRMED')`，不经过 `task.DRAFT`。这保证“AI 建议 ≠ 业务状态变化”。`task.status='DRAFT'` 仅用于用户手动创建后暂存、尚未填写完整必填字段的草稿任务；AI 建议永远先停留在 `ai_suggestion` 中，确认后才创建为 `CONFIRMED` 任务。

客户创建时默认 `LEAD`。V1-A 不自动生成跟进任务，不提供协作、合同或结算状态；阶段变化必须由用户主动发起，并与 `customer_stage_history` 和 `audit_event` 同一事务落库。

| `audit_event` | `(workspace_id, occurred_at DESC)` | 审计追溯。 |

### 5.2 必须在数据库层执行的约束

1. 一用户一空间、授权项唯一、会话 Token 哈希唯一、下载 Token 哈希唯一、活动注销请求唯一。
2. 所有父子资源使用含 `workspace_id` 的复合外键，拒绝跨工作空间关联。
3. 所有状态字段使用 `CHECK` 限定枚举值；状态转换合法性在应用服务事务中校验，关键终态再以条件更新保证。
4. `task.depth` 仅允许 1–3；父子项目/空间一致性用触发器验证。
5. `audit_event` 通过权限和触发器双重保证 append-only。
6. `ai_suggestion` 的终态决议和 `task(source_ai_suggestion_id, source_ai_item_key)` 的唯一性共同保证同一建议条目不能重复落库，同时支持一个建议创建多项任务。

## 6 AI 数据最小化与字段级脱敏

下表"发送给模型时的处理"列指**发送给 LLM 上下文时的脱敏/截断**；主库存储始终为字段原始完整值（受字段长度上限约束），不受本表影响。

| 源表/字段 | 可进入 AI 上下文？ | 发送给模型时的处理 | 说明 |
|---|---|---|---|
| `project.name`、`objective`、`deliverable` | 是，经授权 | 截断（目标/交付物最多 2000 字符，主库仍存 4000 字符） | `AI_BUSINESS_DATA=GRANTED` 才可读取。 |
| `task.title`、`status`、`due_at`、`estimated_minutes` | 是，经授权 | 原枚举/时间，标题截断 200 字符 | 不发送任务完整审计历史。 |
| `customer.name` | 是，经授权且确有能力必要 | token 化，如 `客户-A` | 映射仅存在单次内存上下文；不同 run 之间不复用。 |
| `customer.stage`、`intent_level`、`next_action` | 是，经授权 | 枚举/截断 | `notes` 默认不发送。 |
| `profile.skills`、`entrepreneur_stage`、`business_goal` | 是，经授权 | 截断 | 仅当前请求。 |
| 手机、邮箱、IP、密码、Refresh Token、证件 | 否 | 删除 | 绝不进入 Prompt。 |
| 客户 `notes`、注销原因、完整审计、原始 AI 输入/输出 | 否 | 删除 | 除非未来单独设计、单独授权与法务评审。 |

每个 `ai_run` 必须保存 `data_categories`、`consent_evidence` 与 `input_redaction_method`；其中 `data_categories` 记录本次调用涉及的数据类别（如 `project`、`task`、`customer`）与服务端使用的别名映射摘要（仅类别与数量，不含可逆映射表），`input_redaction_method` 记录每类的脱敏规则版本（`delete` / `tokenize` / `truncate:N`），三者均为脱敏摘要与规则版本，**不是原始敏感数据副本**。

## 7 状态机与转换规则

### 7.1 项目：`project.status`

| 当前状态 | 允许动作 | 目标状态 | 前置条件 / 事务副作用 |
|---|---|---|---|
| `DRAFT` | 启动 | `IN_PROGRESS` | 至少一个 `CONFIRMED` 或 `IN_PROGRESS` 任务；更新项目、审计同事务。 |
| `DRAFT` | 取消 | `CANCELLED` | 用户确认；不得自动取消任务。 |
| `IN_PROGRESS` | 暂停 | `PAUSED` | 用户确认。 |
| `PAUSED` | 恢复 | `IN_PROGRESS` | 用户确认。 |
| `IN_PROGRESS` / `PAUSED` | 标记完成 | `COMPLETED` | 所有非取消任务均 `COMPLETED` 或 `CLOSED`；系统仅提示，不自动关闭。 |
| `IN_PROGRESS` / `PAUSED` | 取消 | `CANCELLED` | 用户确认；显式处理未终态任务。 |
| `COMPLETED` / `CANCELLED` | — | — | 终态；V1-A 不提供归档/恢复。 |

禁止：无任务启动；以 AI 生成成功自动启动或关闭项目；跨工作空间更新。

### 7.2 个人任务：`task.status`

| 当前状态 | 允许动作 | 目标状态 | 前置条件 / 副作用 |
|---|---|---|---|
| `DRAFT` | 确认 | `CONFIRMED` | 手动草稿完成必填校验；AI 建议确认后直接创建 `CONFIRMED` 任务。 |
| `DRAFT` / `CONFIRMED` / `IN_PROGRESS` | 取消 | `CANCELLED` | 有非终态子任务时要求显式逐项处理。 |
| `CONFIRMED` | 开始 | `IN_PROGRESS` | 记录开始审计。 |
| `IN_PROGRESS` | 标记延期 | `IN_PROGRESS` | 设置 `is_deferred=true`，主状态不变。 |
| `IN_PROGRESS` | 完成 | `COMPLETED` | 设置 `completed_at`；可触发项目完成提示。 |
| `COMPLETED` | 关闭 | `CLOSED` | 设置 `closed_at`。 |
| `CLOSED` / `CANCELLED` | — | — | 终态。 |

`is_overdue` 由查询/定时任务按 `due_at < now()` 且状态不在终态计算或同步，不能用它替代状态转换。SRS 中“待确认/被驳回”属于 AI 提案层，见 §7.4，不落为确认前 `task` 业务行。

### 7.3 AI 运行：`ai_run.status`

| 当前状态 | 事件 | 目标状态 | 规则 |
|---|---|---|---|
| `RECEIVED` | 开始处理 | `PROCESSING` | 先校验用户、空间、授权、配额、脱敏。 |
| `PROCESSING` | 关键输入不足 | `WAITING_FOR_INPUT` | 生成澄清建议；不得猜测。 |
| `WAITING_FOR_INPUT` | 用户补充并重新提交 | `RECEIVED` | 创建新 run，`retry_of_run_id` 不适用。 |
| `PROCESSING` | 结构化输出通过 | `GENERATED` | 创建一个或多个 `ai_suggestion`。 |
| `PROCESSING` | 超时/供应商失败 | `TIMED_OUT` / `FAILED` | 可创建一次同输入重试的独立 run。 |
| `PROCESSING` | Schema 仍失败 | `DEGRADED` | 保存明确标记的自然语言建议，不创建业务对象。 |
| `FAILED` / `TIMED_OUT` / `DEGRADED` / `GENERATED` | — | — | 运行终态；建议状态独立继续。 |

禁止“自动修复”或无限重试；同一输入最多一次后台重试。熔断打开时直接创建可解释的 `FAILED` run（错误码 `AI_CIRCUIT_OPEN`），不访问供应商。

### 7.4 AI 建议：`ai_suggestion.status`

| 当前状态 | 动作 | 目标状态 | 事务要求 |
|---|---|---|---|
| `GENERATED` | 展示 | `WAITING_CONFIRMATION` | 写 `VIEWED` 决议/事件（如需要）。 |
| `WAITING_CONFIRMATION` | 编辑 | `WAITING_CONFIRMATION` | 追加 `EDITED` 决议，保留原建议。 |
| `WAITING_CONFIRMATION` | 确认 | `CONFIRMED` | 同一事务：版本校验 → 写最终项目/任务（如适用）→ 写 `CONFIRMED` 决议 → 写审计。 |
| `WAITING_CONFIRMATION` | 拒绝 | `REJECTED` | 追加 `REJECTED` 决议及可选原因；不写业务对象。 |
| `WAITING_CONFIRMATION` | 过期 | `EXPIRED` | 不自动写业务对象。 |
| `WAITING_CONFIRMATION` | 确认事务失败 | `EXECUTION_FAILED` | 先回滚“确认 + 业务写入 + 成功审计”的完整事务；再以独立失败事务保留建议、失败码和 `EXECUTION_FAILED` 决议，提示手动处理。 |
| `CONFIRMED` / `REJECTED` / `EXPIRED` / `EXECUTION_FAILED` | — | — | 终态。 |

`DAILY_TOP3` 建议的确认不创建任务，只记录用户确认/修改的排序选择；`TASK_PLAN` 确认可批量创建任务，但每项均须满足三级层级、负责人和项目归属约束。`EXECUTION_FAILED` 为终态，**不允许自动回到 `WAITING_CONFIRMATION`**；用户须手动创建或修改任务后再发起新的 AI run，避免静默重试掩盖持久性故障。

### 7.5 授权：`consent.status`

| 当前状态 | 动作 | 目标状态 | 规则 |
|---|---|---|---|
| 不存在 | 同意 | `GRANTED` | 创建当前记录，保留 policy/purpose 版本。 |
| `GRANTED` | 撤回 | `REVOKED` | 立即停止新的相应用途；写审计。 |
| `REVOKED` | 重新同意 | `GRANTED` | 更新版本和时间，`version+1`；写审计。 |
| `GRANTED` | 注销 | `REVOKED`（逻辑） | 主库 tombstone 时停用全部非必要处理。 |

`CORE_SERVICE` 是必需处理：不提供“撤回”动作；用户若不同意，应放弃注册或走注销流程。`AI_BUSINESS_DATA` 撤回后，已有项目/任务/客户仍可手动管理，新的 AI run 必须被拒绝。

### 7.6 导出：`export_job.status`

| 当前状态 | 事件 | 目标状态 | 规则 |
|---|---|---|---|
| `REQUESTED` | 入队成功 | `QUEUED` | 写 `async_job` 和审计。 |
| `QUEUED` | Worker 锁定 | `GENERATING` | 工作空间必须仍可读取且未 tombstone。 |
| `GENERATING` | 文件生成成功 | `READY` | 写私有 object key、7 天 `expires_at`、一次性 token。 |
| `GENERATING` | 可重试错误 | `QUEUED` | 最多 3 次，Worker 记录尝试。 |
| `GENERATING` | 不可恢复/超重试 | `FAILED` | 存稳定失败码，不暴露内部错误。 |
| `READY` | 下载令牌消费 | `DOWNLOADED` | `consumed_at` 同事务更新；同 token 不可再下载。 |
| `READY` | 到期 | `EXPIRED` | 删除对象并撤销 token。 |
| `DOWNLOADED` / `FAILED` / `EXPIRED` | — | — | 终态。 |

### 7.7 注销：`deactivation_request.status` 与账户映射

| 当前状态 | 动作/事件 | 目标状态 | `user` / `workspace` 副作用 |
|---|---|---|---|
| 无活动请求 | 用户提交且近 15 分钟重认证 | `REQUESTED` | 创建请求，同时短暂置 `user/workspace=DEACTIVATION_REQUESTED`。 |
| `REQUESTED` | 冻结事务完成 | `GRACE` | `user=DEACTIVATION_GRACE`，`workspace=READ_ONLY`，禁新业务写和 AI。 |
| `GRACE` | 用户撤销 | `REVOKED` | 恢复 `user=ACTIVE`、`workspace=ACTIVE`，写审计。 |
| `GRACE` | 宽限到期且无保留例外 | `TOMBSTONING` | Worker 取得幂等锁。 |
| `TOMBSTONING` | 匿名化成功 | `TOMBSTONED` | `user=TOMBSTONED`、`workspace=TOMBSTONED`，写 tombstone 摘要。 |
| `GRACE` | 设置保留例外 | `RETENTION_HOLD` | 保持只读，展示原因和预计期限。 |
| `RETENTION_HOLD` | 留存义务解除 | `TOMBSTONING` | Worker 继续匿名化。 |
| `REVOKED` / `TOMBSTONED` | — | — | 终态；`REVOKED` 已恢复为正常账户。 |

`REQUESTED` 仅可短暂存在于提交事务内；对用户可见状态从 `GRACE` 开始。宽限期内可登录只读、可撤销，并保留基础数据导出权；不得创建或修改项目、任务、客户，或发起 AI。宽限期届满后不可恢复。

### 7.8 审计事件：追加生命周期

`audit_event` 没有业务更新状态。生命周期仅为：**构造 → 与业务事务同事务 INSERT → 已持久化 → 按合规策略归档**。任何 UPDATE/DELETE 一律由触发器拒绝；写入失败必须导致其对应关键业务事务回滚。

## 8 关键事务边界

| 用例 | 同一 PostgreSQL 事务必须完成 |
|---|---|
| 项目/任务/客户写入 | 乐观锁条件更新或创建 + 对应状态历史（如有）+ `audit_event` + 幂等响应记录。 |
| AI 建议确认 | 建议版本条件更新 + 决议追加 + 项目/任务写入 + `audit_event` + 必要的指标事件。 |
| 授权变更 | `consent` 条件更新 + `audit_event`；撤回 AI 授权后后续 run 校验立即生效。 |
| 导出申请 | `export_job(REQUESTED/QUEUED)` + `async_job(QUEUED)` + `audit_event` + 幂等记录。 |
| 注销申请 | `deactivation_request` + `app_user/workspace` 只读状态 + 撤销除当前 Session 外的其他 Session family + AI 拒绝标记 + `async_job` + `audit_event`；当前 Session 仅允许只读/导出/撤销。 |
| 任务完成 | 条件更新 + `audit_event`；只计算“可提示项目完成”，不得自动关闭项目。项目完成提示为内存态/查询态（`SELECT count(*) FROM task WHERE project_id=? AND status NOT IN ('COMPLETED','CLOSED','CANCELLED')`），**不写入 `project` 表的 hint 字段**，避免与项目状态机耦合。 |

外部副作用（对象存储、模型调用、邮件）不得置于长数据库事务内。事务先持久化可追踪 `async_job` / `ai_run`，Worker 以幂等键执行；若外部成功而回写失败，依靠 job 重试和资源状态查询收敛。

## 9 迁移与发布计划

### 9.1 迁移原则

- 所有结构变更通过版本化 SQL 迁移；不手工改生产库。
- 使用 expand/contract：先新增可空字段/索引/双写，再回填、切读、最后删除旧字段。
- 破坏性变更必须有可测试回滚脚本；分区、触发器、权限变更也必须进入迁移版本。
- 每份迁移在预发执行：schema 校验、权限校验、至少 10 万条模拟审计事件分区写入、回滚演练。
- `audit_event` 月分区维护：M05 迁移时预创建当前月及未来 12 个月分区；后续由定时 Worker（`async_job.job_type='AUDIT_PARTITION_MAINTAIN'`，V1-A 可复用现有 Worker 池）每月执行 `CREATE TABLE IF NOT EXISTS audit_event_yYYYYmm ... PARTITION OF audit_event FOR VALUES FROM ...` 并冷迁移超过 13 个月的分区到归档表空间（V1-A 暂用同实例独立表空间，V1-B 评估外部不可变存储）。预创建失败需告警，但不阻塞业务写入（写入无对应分区会报错，故须在月初之前完成）。

### 9.2 建议迁移批次

| 批次 | 内容 | 上线门槛 |
|---|---|---|
| M01 | 扩展、通用函数、`app_user`、`credential`、`session`、`workspace`、`profile`、`platform_operator`、`platform_operator_role` | 普通用户注册/登录/刷新、一用户一空间、运营独立入口和 MFA/角色集成测试通过。 |
| M02 | `project`、`task`、`customer`、`customer_stage_history`、复合外键与任务深度触发器 | 三级任务、越权、乐观锁测试通过。 |
| M03 | `consent`、`ai_run`、`ai_suggestion`、`ai_suggestion_decision`、`ai_usage_daily` | AI 授权、Schema、确认无自动写入测试通过。 |
| M04 | `export_job`、`export_download_token`、`deactivation_request`、`async_job`、`idempotency_record` | Worker 幂等、导出单次下载、注销恢复测试通过。 |
| M05 | `audit_event` 分区、`audit_owner`、`append_audit_event` 函数、append-only 触发器、`metric_daily_aggregate` | 任意 UPDATE/DELETE 审计事件失败；业务/Worker 无直接表写权限；所有关键写入均同事务留痕。 |

### 9.3 初始数据与配置

迁移只初始化不可变枚举约束和必要授权项定义；不在数据库硬编码模型密钥、供应商 endpoint、可调整的配额。可调整 AI 限流值由受控配置/KMS 引用提供，配置变更写 `audit_event`。

## 10 注销、留存、备份与归档

| 数据 | 主库策略 | 默认留存 / 删除规则 |
|---|---|---|
| 用户、画像、项目、任务、客户 | 宽限期届满后匿名化；保留不可反查主键和最小 tombstone | 注销后 30 天宽限；保留例外单独记录。 |
| Session / Refresh Token | 注销申请时撤销除当前 Session 外的其他会话；当前 Session 保留至退出/过期，仅可只读、导出和撤销注销；tombstone 时全部撤销 | 按会话到期和安全留存策略。 |
| AI run / suggestion | 不保留原始 Prompt/输出；保留脱敏摘要、模型版本、授权证据 | 默认 90 天，法务确认后可调整。 |
| Audit event | append-only，归档而非随业务级联删除 | 热数据至少 13 个月；安全事件和管理员操作按法务要求延长。 |
| Export 文件/令牌 | `READY` 7 天或消费后清理对象并撤销 token | 不保留公开 URL。 |
| 备份 | 不主动改写历史备份；tombstone 后新备份只含匿名形态 | 最长按 30 天自然覆盖；恢复演练不得使已注销用户重新对外可见。 |

> **法务确认项**：13 个月为 V1-A 默认策略，应用日志 180 天、AI 脱敏摘要 90 天、备份 30 天自然覆盖；法务若要求不同期限，应通过 ADR 修订留存作业，不修改 append-only、最小化和注销匿名化原则。

## 11 API、Worker 与测试交接

### 11.1 05 API 契约必须表达

1. 资源写请求必传 `expectedVersion`（创建除外）与 `Idempotency-Key`；响应返回最新 `version`。
2. 所有资源路径从认证上下文派生工作空间，客户端不得提交决定权限边界的 `workspace_id`。
3. AI 确认接口传 `suggestionId + expectedVersion + editedPayload`，不允许客户端提交任意项目/空间归属。
4. 导出下载接口只接受一次性令牌；令牌消费成功后不可重放。
5. 注销申请接口要求 15 分钟内重认证证明；只读状态返回稳定业务错误码。

### 11.2 Worker 必须实现

- `EXPORT_GENERATE`：以创建时空间归属生成 CSV；不在导出中混入别的空间数据。
- `AI_RETRY`：最多一次同输入重试；不得重复创建建议或任务。
- `DEACTIVATION_FINALIZE`：复核 `grace_ends_at`、`retention_hold`、版本和幂等键后才匿名化。
- `EXPORT_CLEANUP`：删除到期对象、撤销下载令牌并记录安全审计。

### 11.3 数据与状态机验收最小集

1. 任意 ID 篡改、跨空间复合外键、裸主键仓储调用均不能读取/写入他人资源。
2. 每个状态机穷举合法/非法转换；非法转换不改变版本、不写成功审计。
3. 同一建议双击确认只能创建一次业务对象；同一幂等键重复请求返回相同结果。
4. 审计表 UPDATE/DELETE 必须被 DB 触发器拒绝；业务/Worker 无直接 INSERT 权限，只能通过受控 `append_audit_event` 函数在业务事务中追加。
5. AI 授权撤回后，新 AI run 被拒绝，但手动项目/任务 CRUD 仍可用。
6. 注销宽限期只读、撤销恢复、留存例外、tombstone、备份自然覆盖均可演练。
7. AI 上下文和审计摘要均不得含密码、令牌、原始证件、客户完整备注或原始 Prompt/输出。
