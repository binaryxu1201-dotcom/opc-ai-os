## 1 目标、范围与质量原则

### 1.1 目标

本计划将 V1-A 的业务、数据、API、AI、安全、可用性和 UI/UX 约束转化为可执行的测试层级、环境、发布门禁、回滚、备份恢复和运营观测流程。通过本计划是 V1-A 面向真实用户发布的必要条件，不等同于进入 V1-B 的阶段门通过。

### 1.2 测试范围

覆盖：注册/登录/会话族、工作空间/画像、项目/三级任务/CRM、AI COO、授权、导出、注销、审计、运营聚合、PWA 响应式与无障碍、数据迁移、Worker、部署与灾备。

不覆盖：多成员协作、协作市场、支付/结算、站内沟通、文件服务、内容实验、RAG/长期记忆、AI CLO、企业实名等 V1-A 非目标。测试中发现非目标能力入口、数据表、API 或隐式功能时，按范围泄漏缺陷处理。

### 1.3 质量原则

1. **先阻断高风险**：越权、AI 自动写入、认证/会话、注销、导出令牌、审计缺失、数据泄露均为发布阻断项。
2. **服务端事实优先**：测试不得只断言 UI 文案；关键流程必须验证 API 响应、数据库状态、审计事件和 Worker 副作用一致。
3. **确定性与 AI 分离验证**：项目/任务/客户手动路径必须独立于模型服务通过；AI 质量/安全另设测试集和门槛。
4. **真实故障可降级**：模拟 Redis、模型、对象存储、邮件和数据库依赖故障，验证经营核心功能与用户可理解反馈。
5. **最小化生产数据**：开发、测试、预发禁止使用生产个人数据；性能、恢复与 AI 质量使用合成或获授权的脱敏数据。
6. **证据可追溯**：每个门禁保存测试报告、版本、环境、时间、样本、失败缺陷和批准人；不得以口头结论代替记录。

## 2 质量目标与发布口径

### 2.1 发布质量门槛

| 维度 | V1-A 发布门槛 | 测量口径 |
|---|---|---|
| 核心 E2E | 通过率 100% | §5.1 核心旅程全部通过，无 blocker/critical 缺陷。 |
| 单元测试 | 核心业务模块行/分支覆盖率 ≥60% | Auth、Workspace、Project/Task、AI confirm、Consent/Lifecycle、Audit。 |
| API 契约 | 核心契约通过率 100% | §5.2 所列认证、隔离、幂等、状态和错误码用例。 |
| AI 质量 | 任务拆解 ≥100 样本，整体通过率 ≥90% | §6 的自动检查 + 人工抽检汇总。 |
| 安全 | 高危漏洞 0；越权高危问题 0；重大安全/合规事件 0 | SAST/SCA、越权矩阵、渗透检查、发布事件台账。 |
| 性能 | §7 所有 P95 达标 | Edge 到响应返回的端到端压测。 |
| 可靠性 | RPO ≤24h、RTO ≤4h 演练通过 | §8 备份恢复记录。 |
| 合规 | 授权、导出、注销、审计、AI 数据边界书面确认 | 法务/安全检查表签署。 |
| 可观测性 | 关键日志/指标/告警可触发并关联 | §9 演练通过。 |

### 2.2 缺陷分级与决策

| 级别 | 定义 | 发布处理 |
|---|---|---|
| Blocker | 数据泄露、越权、AI 自动业务写入、不可恢复数据损失、认证失效、核心旅程不可用。 | 阻断发布；修复后完整回归。 |
| Critical | 核心流程失败、审计缺失、注销/导出错误、重大性能未达标、严重合规风险。 | 阻断发布；需负责人复验。 |
| Major | 有明确替代路径但明显影响主要体验/指标。 | 默认阻断；产品、技术、测试共同书面豁免才可带缺陷发布。 |
| Minor | 文案、视觉、非核心边缘场景问题，无安全/数据风险。 | 可带缺陷发布，进入下一迭代并标记负责人/截止日期。 |

任何 Blocker/Critical 不允许以功能开关、前端隐藏或运营流程绕过替代修复，除非该能力可被完全关闭且核心手动经营路径仍符合范围与验收。

## 3 环境、数据与工具

### 3.1 环境矩阵

| 环境 | 用途 | 数据与外部依赖 | 发布规则 |
|---|---|---|---|
| 本地开发 | 单元、组件、开发调试 | 合成数据；本地 PostgreSQL/Redis；模型 Mock 默认开启。 | 不作为质量证据。 |
| CI | 静态、单元、契约、迁移检查 | 临时容器数据库/Redis；模型 fixture/mock。 | 每次 PR/合并必须通过。 |
| 测试 | 集成、E2E、状态机、Worker | 固定合成种子数据；受控模型测试配置。 | 每日自动回归，缺陷可复现。 |
| 预发 | 性能、安全、发布演练 | 不使用生产 PII；模型使用经合同审查的沙箱/限额配置；独立 KMS 密钥。 | 与生产拓扑、迁移、权限、网关配置尽量一致。 |
| 生产 | 灰度与真实运营 | 中国大陆存储；受控生产密钥与审计。 | 仅通过所有发布门禁后部署。 |

### 3.2 测试数据规则

1. 每个测试工作空间使用可辨识合成前缀，如 `test-{suite}-{runId}`，便于清理和阶段门指标排除。
2. 禁止在日志、截图、fixture、压测脚本或 AI 测试样本中写入真实密码、令牌、身份证件、完整合同、未授权私聊或客户完整备注。
3. AI 测试样本以人工编写的合成项目描述构成；如使用真实案例，必须先获得书面授权并经脱敏审查。
4. 测试/预发的导出文件、备份和对象存储生命周期不超过 7 天，独立加密并按 run ID 清理。
5. 测试账户、运营账户、压测流量在埋点中带 `is_test_traffic=true`，所有阶段门指标查询必须排除它们。

### 3.3 建议工具与证据

| 类别 | 建议实现 | 产物 |
|---|---|---|
| 单元/集成 | TypeScript 测试框架 + Testcontainers PostgreSQL/Redis | JUnit/覆盖率报告。 |
| API/契约 | OpenAPI/JSON Schema fixture + HTTP 测试客户端 | 契约报告、请求/响应脱敏样本。 |
| E2E/可访问性 | Playwright + axe-core 或等价方案 | 视频/截图、trace、无障碍报告。 |
| 性能 | k6、Artillery 或等价压测工具 | P50/P95/P99、吞吐、错误率和资源利用率报告。 |
| 安全 | SAST、依赖/SBOM 扫描、DAST 基础扫描、人工越权/Prompt 注入用例 | 扫描报告、例外审批。 |
| AI 质量 | JSON Schema 校验器 + 标注表/评审工具 | 样本集版本、评分明细、聚合通过率。 |
| 可观测性 | 结构化日志查询、指标仪表盘、告警演练 | requestId/traceId 样例、告警通知记录。 |

工具可替换，但不能削弱以下验证能力：真实 PostgreSQL 事务/触发器、Redis 过期行为、SSE 断线、Worker 幂等、移动浏览器 E2E、P95 压测、备份恢复及审计权限验证。

## 4 测试层级与 CI 流程

### 4.1 测试层级

| 层级 | 目标 | 最小覆盖 |
|---|---|---|
| 静态检查 | 类型、格式、依赖与秘密 | TypeScript 类型、lint、格式、提交密钥扫描、许可证/SBOM。 |
| 单元测试 | 领域规则和纯函数 | 状态机、授权判定、脱敏、Token 轮换判定、配额、指标口径。 |
| 集成测试 | 与真实基础设施的边界 | PostgreSQL 约束/触发器/分区、Redis 限流、对象存储 adapter、队列 Worker。 |
| API 契约测试 | HTTP/SSE 行为稳定 | 状态码、错误码、JSON Schema、Header、分页、幂等、版本冲突。 |
| E2E | 用户核心旅程 | 移动/桌面、PWA、表单、错误状态、重认证、注销。 |
| 安全/隐私 | 攻击面和数据边界 | CSRF/Origin、越权、Token、Prompt 注入、敏感数据泄露。 |
| 性能/可靠性 | SLO、降级、恢复 | P95、限流、熔断、重试、备份恢复、迁移回滚。 |
| AI 质量 | 有用、可控、可追溯 | 结构化输出、澄清、编辑确认、事实边界、样本通过率。 |

### 4.2 CI 门禁

| 时点 | 必须执行 | 失败处理 |
|---|---|---|
| 每次提交/PR | 类型检查、lint、单元、依赖/秘密扫描 | 阻止合并。 |
| 合并到主干 | 核心 API 契约、集成、迁移检查、权限回归、AI JSON Schema fixture | 阻止部署到测试环境。 |
| 每日 | 测试环境核心 E2E、Worker 回归、AI 小样本冒烟、移动浏览器回归 | 建立缺陷；核心失败阻止下次发布候选。 |
| 发布候选 | 全量 E2E、性能、越权、Prompt 注入、AI 质量、备份恢复、迁移回滚、无障碍 | 任一发布门禁失败即拒绝发布。 |

### 4.3 数据库迁移测试流程

每份迁移按以下顺序验证：

1. 在空库应用全部迁移，验证 schema、索引、角色、触发器、分区创建。
2. 在前一版本合成数据快照上升级，验证数据回填、应用兼容和查询性能。
3. 针对 M01–M05 分别执行对应门槛：会话/空间、三级任务、AI 确认、导出/注销、审计 append-only。
4. 演练回滚：应用代码回滚时，新迁移字段仍能被旧版代码忽略；破坏性结构变更仅可在 expand/contract 的 contract 阶段执行。
5. 生成迁移审查记录，含 SQL diff、锁表风险、预计执行时间、回滚脚本和审批人。

## 5 功能、契约与 E2E 测试矩阵

### 5.1 核心端到端旅程

| 编号 | 旅程 | 核心断言 | 自动化 |
|---|---|---|---|
| E2E-01 | 注册→工作空间→画像→工作台 | 3 分钟内完成；注册审计记录条款/隐私版本；创建空间时才创建 `CORE_SERVICE`；未完成画像不能创建项目。 | 桌面 + 移动。 |
| E2E-02 | 手动创建项目与三级任务 | 项目初始 `DRAFT`；任务深度不超过 3；负责人固定空间所有者。 | 桌面 + 移动。 |
| E2E-03 | 项目/任务状态转换 | 合法路径成功；非法路径 409；项目不被自动完成。 | API + E2E。 |
| E2E-04 | AI 任务拆解→编辑→确认 | AI 未确认前无 `task` 行；确认后创建 `CONFIRMED` 任务，原建议/决议/审计可追溯。 | Fixture 模型 + DB 断言。 |
| E2E-05 | AI 失败降级 | 超时/Schema 失败/熔断时可手动创建项目和任务；不得自动写业务对象。 | Mock 依赖。 |
| E2E-06 | 每日三件事 | 最多 3 项；无建议时显示活跃任务降级；确认不创建/修改任务。 | 桌面 + 移动。 |
| E2E-07 | CRM-lite | 创建客户初始 `LEAD`；仅允许阶段矩阵路径；历史和审计同事务追加。 | API + E2E。 |
| E2E-08 | 授权撤回 | 撤回 AI 授权后新 AI run 403；手动经营继续可用；审计存在。 | API + E2E。 |
| E2E-09 | 导出与一次性下载 | 生成 CSV 仅含当前空间；token 一次消费；过期/重放处理正确。 | Worker + API。 |
| E2E-10 | 注销宽限与撤销 | 重认证后进入 `GRACE`、空间只读；除当前 Session 外全部撤销，当前 Session 仍可读/导出/撤销；撤销恢复 `ACTIVE`。 | API + E2E。 |
| E2E-11 | 注销留存与 tombstone | 通过测试环境受控时钟/调度推进 30 天宽限期；验证 `RETENTION_HOLD` 不可恢复、到期后 Worker 匿名化、审计留痕和备份规则符合 §8。不得依赖真实等待。 | Worker + DB。 |
| E2E-12 | 会话轮换与失效 | Refresh 轮换；失效 Token 重放撤销 session family；客户端收到 419 后重新登录。 | API + 浏览器。 |
| E2E-13 | 运营聚合与配额 | 指标仅聚合；运营端不能读取私密业务；配额变更留审计并不影响手动业务。 | API + 角色测试。 |

### 5.2 API、并发与数据完整性矩阵

| 编号 | 场景 | 必须断言 |
|---|---|---|
| API-01 | 认证和安全 Header | 所有写 API 缺 `Origin` 或 `X-OPC-CSRF` 均 403；CORS 非白名单拒绝，禁止 `*`。 |
| API-02 | Access/Refresh Token | Access 过期 401；Refresh 重放/撤销/过期 419；Refresh Cookie 不出现在 JSON/日志。 |
| API-03 | 工作空间隔离 | URL/ID/分页 cursor/批量 ID/导出/异步任务/运营接口的跨空间访问均 404 或拒绝，不泄露存在性。 |
| API-04 | 幂等 | 同 `Idempotency-Key` + 同请求返回首次结果；同键不同请求 409；Redis 故障不影响 DB 幂等事实。 |
| API-05 | 乐观锁 | 旧 `expectedVersion` 409；不写业务状态、不写成功审计；批量操作按行返回。 |
| API-06 | 状态机 | 穷举项目、任务、客户阶段、AI run/建议、授权、导出、注销的合法/非法转换。 |
| API-07 | 导出 token | POST body token 不进 URL/Referer/日志；只允许颁发用户下载；消费后重放 409。 |
| API-08 | 重认证 | 修改密码、敏感授权、注销/撤销注销在当前 Session 重认证 >15 分钟后返回 422 `REAUTH_REQUIRED`；另一设备刚重认证也不能放行本 Session。 |
| API-09 | SSE | `Last-Event-ID` 重连可恢复持久状态；断线不取消 run；事件不含原始 Prompt/token/未脱敏客户数据。 |
| API-10 | 错误契约 | 400/401/403/404/409/419/422/429/503 的稳定码、中文文案安全性和 `requestId` 一致。 |
| API-11 | 运营用户搜索 | 少于 4 字符拒绝、严格限流、返回掩码联系方式且不含业务内容；每次搜索写管理员审计。 |

### 5.3 数据库与 Worker 矩阵

| 编号 | 场景 | 必须断言 |
|---|---|---|
| DB-01 | 复合外键与唯一约束 | 不能把项目、任务、客户、AI 建议、导出关联到其他 `workspace_id`。 |
| DB-02 | 三级任务 | 第 4 层任务、跨项目父任务、父子深度不连续均被触发器/约束拒绝。 |
| DB-03 | 审计不可变与原子性 | 业务/Worker 无直接表写权限；仅受控 `append_audit_event` 可执行；任何 UPDATE/DELETE 均被触发器拒绝；业务事务回滚时审计也回滚。 |
| DB-04 | AI 确认事务 | 一条建议可用不同稳定 `itemKey` 创建多项任务；同一 `(suggestionId,itemKey)` 不可重复。成功时建议/决议/任务/审计同时存在；失败时业务写与成功审计回滚，仅留 `EXECUTION_FAILED` 记录。 |
| DB-05 | 导出 Worker | 至少一次投递不产生多份业务结果；最多 3 次重试后 `DEAD_LETTER` 和告警。 |
| DB-06 | 注销 Worker | `locked_at` + 幂等键防止重复 tombstone；`retention_hold` 阻止自动匿名化。 |
| DB-07 | 审计分区 | 当前/未来 12 月分区存在；月度维护失败告警；10 万条模拟写入和查询符合性能预期。 |
| DB-08 | 迁移回滚 | M01–M05 升级/代码回滚不丢数据；触发器/权限/分区均可审查和回退。 |

### 5.4 UI/UX 与可访问性矩阵

| 编号 | 场景 | 必须断言 |
|---|---|---|
| UX-01 | 响应式 | 360px、768px、1440px 下导航、表单、任务树、AI 建议与粘性操作区可用。 |

## 6 AI 质量、安全与成本测试

### 6.1 AI 测试集与版本管理

| 测试集 | 最小规模 | 覆盖 | 发布要求 |
|---|---:|---|---|
| 任务拆解标准集 | ≥100 | 行业、复杂度、目标明确度、不同任务层级。 | 整体通过率 ≥90%。 |
| 每日三件事集 | ≥50 | 活跃/暂停项目、截止日期、任务状态、空任务。 | 排序正确、最多三项、无任务降级全部通过。 |
| 低质量输入集 | ≥20 | 信息不足、矛盾、含混、多义。 | 返回澄清/安全通用建议，不编造业务事实。 |
| Prompt 注入集 | ≥30 | 忽略系统指令、越权 workspace、泄露 Prompt、工具滥用、数据外传诱导。 | 成功攻击率 0；无跨空间/敏感泄露/自动写。 |
| 脱敏集 | ≥30 | 手机、邮箱、客户名、客户备注、证件、令牌、注入文本。 | 禁止字段不发送；允许字段按规则 token 化/截断。 |

每个测试集均版本化，记录样本 ID、输入分类、预期结果、模型/Prompt 版本、实际结果、自动评分、人工评分、评审人和日期。模型或 Prompt 变更必须重跑全量任务拆解集、低质量输入集、Prompt 注入集和脱敏集。

### 6.2 任务拆解评分规则

| 指标 | 门槛 | 评分方法 |
|---|---:|---|
| 不改变项目目标 | ≥95% | 人工抽检/标注比较。 |
| 生成至少 3 项可执行任务 | ≥90% | 自动结构校验。 |
| 每项具负责人、截止时间或待确认字段 | ≥95% | JSON Schema + 规则检查。 |
| 无明显重复/矛盾 | ≥95% | 人工抽检。 |
| 用户 3 次内可编辑/确认 | ≥80% | 线上观测，作为阶段门观察指标，不阻断首次发布。 |
| 结构化输出可追溯 | 100% | 每个 run 有 traceId、模型/Prompt 版本、授权依据和摘要。 |

发布 AI 质量总分以关键硬失败为一票否决：任意样本出现跨工作空间数据、敏感字段泄露、自动创建/修改/删除业务对象，或明显虚构具体客户/任务事实，立即判失败并阻断发布，即使总体通过率 ≥90%。

### 6.3 AI 失败、降级与成本

| 场景 | 注入方式 | 必须断言 |
|---|---|---|
| 首 token 超时 / 总时长 >15s | 模型 adapter 延迟 | UI 可离开、run 可回查、最多一次同输入重试、手动路径可用。 |
| 无效 JSON / Schema 失败 | 返回非法/缺字段 JSON | 一次重试后 `DEGRADED`；仅自然语言建议，无业务写。 |
| 供应商 5 分钟失败率 ≥30% | 故障 Mock | 熔断 60 秒、探针恢复、返回 `AI_CIRCUIT_OPEN`。 |
| 用户/能力/全局配额 | 预置计数超限 | 429 稳定错误码，手动业务不受影响。 |
| 成本异常 | 伪造 token 成本超阈值 | 告警触发；管理员可关闭能力，历史业务不受影响。 |

## 7 性能、容量与安全测试

### 7.1 性能测量口径

| 指标 | 目标 | 起止点 | 负载与通过标准 |
|---|---:|---|---|
| 关键首屏 | P95 ≤2s | 浏览器导航开始至关键工作台内容可交互。 | 4G 模拟移动网络与桌面宽带分别测量。 |
| 查询 API | P95 ≤500ms | Edge 接收请求至完整响应返回，排除客户端渲染。 | 1k 并发目标下核心列表/详情；错误率 <1%。 |
| 普通写 API | P95 ≤300ms | Edge 接收至业务事务+审计提交响应。 | 项目/任务/客户/授权写；错误率 <1%。 |
| AI 首反馈 | P95 ≤3s | 用户触发到首个 SSE `run.created`/可消费反馈。 | 受控模型延迟与真实沙箱各一轮。 |
| AI 任务生成 | P95 ≤15s | 创建 run 到 `suggestion.ready`/终态（含降级）。 | 任务拆解典型输入；超时路径单独统计。 |

压测脚本必须标注测试账户、独立 workspace、幂等键和可清理资源。不得用同一账户制造全部并发，避免掩盖会话、配额、隔离或锁冲突问题。

### 7.2 负载场景

| 场景 | 配比 | 断言 |
|---|---|---|
| 经营读写混合 | 70% 查询、20% 普通写、10% AI/导出触发 | 核心 P95、连接池、数据库 CPU、错误率、审计写入延迟。 |
| AI 峰值保护 | 并发触发任务拆解/每日三件事 | 用户/能力/全局限流生效，系统不耗尽 DB/Worker。 |
| 队列积压 | 暂停 Worker 后批量发起导出/AI 重试 | API 仅创建 job、队列告警、恢复后幂等清空。 |
| 大审计分区 | 10 万条模拟审计写入/查询 | 月分区路由正确、查询索引命中、维护不影响业务写。 |

### 7.3 安全与隐私矩阵

| 编号 | 测试 | 必须断言 |
|---|---|---|
| SEC-01 | HTTPS/HSTS/CORS | 仅 HTTPS；HSTS；CORS 精确白名单，无 `*`。 |
| SEC-02 | CSRF/Origin | 写操作缺/伪造 Header/Origin 全部拒绝；登录/刷新同样受保护。 |
| SEC-03 | 鉴权/越权 | ID/URL/cursor/批量/下载/Worker/运营端越权不成功，返回不泄露资源的错误；运营接口必须使用独立 `platform_operator` 身份并完成 MFA。 |
| SEC-04 | Refresh 重放 | 旧 Refresh 使用后撤销整个 family，所有设备重登；日志不含 token。 |
| SEC-05 | 输入与注入 | SQL/NoSQL/XSS/路径/JSON body 超限等输入被验证、编码或拒绝。 |
| SEC-06 | 密钥与日志 | KMS 引用、无硬编码密钥；结构化日志不含密码/令牌/PII/原始 Prompt。 |
| SEC-07 | AI Prompt 注入 | §6.1 注入集全通过；工具仅只读、仅当前空间、无通用执行。 |
| SEC-08 | 数据最小化 | 模型请求捕获中不存在禁止字段；客户名按 alias、备注不发送。 |
| SEC-09 | 审计 | 写操作都有审计；摘要字段白名单生效；审计表无 UPDATE/DELETE；业务事务回滚时审计同步回滚；应用日志/AI 摘要按留存策略清理。 |
| SEC-10 | 依赖漏洞 | 高危依赖、镜像和 IaC 漏洞为 0；例外需安全批准与到期日。 |

## 8 可靠性、备份、恢复与回滚

### 8.1 故障降级演练

| 故障 | 注入 | 通过标准 |
|---|---|---|
| LLM 不可用 | 503/超时/熔断 | 手动项目/任务/客户可用；AI 返回可读降级信息。 |
| Redis/队列不可用 | 停止 Redis 或拒绝队列连接 | 同步业务写继续基于 PostgreSQL；异步功能提示延迟；不把 Redis 当事实源。 |
| Worker 停止 | 暂停 Worker | 导出/AI 重试/注销状态可追踪；队列积压告警；恢复后幂等处理。 |
| 对象存储不可用 | 导出上传失败 | job 重试/失败码正确；不泄露 object key；核心业务继续。 |
| 数据库短暂连接失败 | 注入连接超时 | 健康检查失败、有限重试、用户安全错误；不写半事务。 |

### 8.2 备份与恢复演练

发布前至少完整演练一次，季度至少一次复演：

1. 选择预发合成数据快照，验证备份加密、访问权限和完整性校验。
2. 在与生产等价安全控制的隔离恢复环境中恢复数据库、审计分区、对象存储导出元数据和必要配置引用；禁止把生产用户数据复制至低安全环境。
3. 记录恢复开始/结束时间，验证 RTO ≤4h；核对恢复点与备份时间，验证 RPO ≤24h。
4. 抽样验证用户/空间/项目/任务/客户、授权、AI 摘要、导出、审计、会话撤销和 Worker 作业的一致性。
5. 注销场景：恢复包含 tombstone 的快照时，已匿名化主体不得重新对外可见；历史备份按自然 30 天保留期覆盖，不主动改写。
6. 生成演练记录、差异清单、耗时、负责人和结论；任一 RPO/RTO 或匿名化检查失败阻断发布。

### 8.3 回滚策略

| 触发条件 | 立即动作 | 回滚路径 |
|---|---|---|
| 核心 API 错误率/延迟异常 | 停止扩量、切换上一应用版本。 | 应用蓝绿/滚动回滚；数据库保持 expand 阶段兼容。 |
| AI 质量/供应商故障 | 关闭 AI 能力开关。 | 保留项目/任务/CRM；Prompt/模型版本回退，不回滚业务数据。 |
| 安全事件/越权迹象 | 关闭受影响入口、撤销会话族、保全审计、启动事件响应。 | 修复后安全复验；不得删改审计。 |
| 迁移异常 | 暂停发布、只执行已演练回滚脚本。 | 代码回退优先；破坏性 schema 不做即时删除，必要时恢复至备份。 |
| Worker 积压/重复执行 | 停止消费者、检查幂等/锁、恢复限速消费。 | 不人工重复投递未知状态 job。 |

生产回滚必须保留：触发指标、时间线、影响范围、执行人、版本、审计/日志关联、用户沟通与事后改进项。

## 9 观测、告警与阶段门运营

### 9.1 发布前可观测性检查

| 类别 | 必须存在 | 验证 |
|---|---|---|
| 关联标识 | `requestId`、AI/Worker `traceId` | 从 HTTP → AI run/job → 审计事件可关联抽样 10 条。 |
| 平台健康 | 请求量、P95/P99、错误率、DB 连接、Redis、队列积压、缓存命中 | 仪表盘查询可用，阈值可配置。 |
| AI 质量 | Schema 失败、超时、熔断、编辑/驳回、成本、结构化成功率 | 与 `ai_run` / `ai_usage_daily` 可核对。 |
| 安全 | 登录锁定、限流、鉴权拒绝、Prompt 拦截、会话重放 | 脱敏日志能检索，不能暴露 PII。 |
| 生命周期 | 导出成功率、token 消费失败、注销到期积压、备份成功 | Worker 状态和告警可验证。 |

### 9.2 告警与响应

| 告警 | 初始阈值 | 响应时限 | 第一处置 |
|---|---|---|---|
| 核心 API 5xx | 5 分钟 >2% | 15 分钟 | 暂停发布/扩量，检查最近变更和依赖。 |
| AI 失败/超时 | 5 分钟 >30% | 15 分钟 | 检查熔断，必要时关闭 AI 能力。 |
| 队列积压 | 超过 15 分钟无下降 | 30 分钟 | 检查 Worker、限流和死信。 |
| 数据库不可用 | 健康检查连续 2 次失败 | 5 分钟 | 启动数据库故障流程，停止写入扩散。 |
| 备份失败 | 任一每日备份失败 | 4 小时 | 重新执行并调查；连续失败阻断后续发布。 |
| 高危安全事件 | 任一命中 | 立即 | 关闭受影响入口、保全审计、通知安全负责人。 |
| 注销到期积压 | 到期后 >24h 未完成 | 4 小时 | 检查 retention hold、Worker 和幂等锁。 |

### 9.3 阶段门监测

V1-A 上线后，阶段门不是一次发布检查，而是最少 100 名有效激活用户、连续至少 2 周的运营观察。指标固定沿用 SRS §6.7 / PRD §1.6 口径，测试/运营/压测流量必须排除。

| 指标 | 必须达标 | 固定口径 |
|---|---:|---|
| 注册至首次创建项目转化率 | ≥50% | 注册后 7 天内发生 `project_created`。 |
| AI 任务建议确认率 | ≥30% | `ai_suggestion_confirmed` / 已展示建议数，含编辑后确认。 |
| 激活用户 7 日留存 | ≥35% | 激活后第 7–13 天至少一次有效登录。 |
| 每日三件事周活跃率 | ≥40% | 激活用户每周至少一次 `daily_top3_viewed`。 |
| AI 质量测试集通过率 | ≥90% | §6 发布前样本结果。 |
| 重大安全/合规问题 | 0 | 安全与运营事件台账。 |

## 10 发布流程与门禁

### 10.1 发布前检查清单

| 门禁 | 证据 | 责任角色 | 必须通过 |
|---|---|---|---|
| G0 范围冻结 | 非目标扫描、API/表/路由清单 | 产品 + 技术 | 是 |
| G1 代码质量 | CI 静态/单元/覆盖率/依赖扫描 | 开发负责人 | 是 |
| G2 数据与迁移 | M01–M05 迁移、约束、触发器、回滚报告 | 后端/DB 负责人 | 是 |
| G3 核心功能 | E2E-01 至 E2E-13 100% | 测试负责人 | 是 |
| G4 API/安全 | §5.2、§7.3、越权/Prompt 注入报告 | 安全 + 测试 | 是 |
| G5 AI 质量 | §6 测试集、样本评分、模型/Prompt 版本 | AI 负责人 + 产品 | 是 |
| G6 性能与可靠性 | §7 压测、§8 故障/恢复演练 | SRE/技术负责人 | 是 |
| G7 合规与隐私 | 授权/导出/注销/审计/模型合同检查表；DMD-02 运营管理员清册与 MFA；DMD-03 留存期限书面确认 | 法务 + 安全 | 是 |

| G9 运营就绪 | 仪表盘、告警、Runbook、值班和阶段门口径 | 运营 + SRE | 是 |

### 10.2 灰度发布步骤

1. **T-7 天**：冻结发布候选，完成 G0–G5，修复 Blocker/Critical。
2. **T-3 天**：预发执行 G6–G8，完成备份恢复、迁移回滚、性能和安全复验。
3. **T-1 天**：发布评审会审阅全部证据，确认 G9、回滚负责人、值班表与用户沟通模板。
4. **T0**：部署应用与兼容迁移；先启用内部测试账户，进行 30 分钟冒烟。
5. **T0 + 30 分钟**：若核心指标稳定，灰度至 5% 受邀用户；AI 能力默认受配额和熔断保护。
6. **T0 + 4 小时**：无告警/Blocker 后扩至 25%；复核错误率、P95、队列、AI 成功率、审计和导出。
7. **T0 + 24 小时**：无回滚条件后扩至 100% 目标用户；进入 2 周阶段门观察。

灰度比例可按实际用户规模调整，但每次扩量前必须保存观测快照，确认 §9.2 告警均未触发且关键旅程样本可用。

### 10.3 发布停止与回滚条件

任一项命中即暂停扩量；满足严重条件时立即回滚/关闭能力：

| 条件 | 动作 |
|---|---|
| 任意越权成功、敏感数据进入模型/日志、审计被修改 | 立即关闭受影响能力，启动安全事件响应，阻断继续发布。 |
| 项目/任务/客户核心写失败率 >1% 持续 5 分钟 | 停止扩量，回滚应用版本。 |
| 核心 API 5xx >2% 持续 5 分钟 | 停止扩量，排障或回滚。 |
| 查询/写 P95 超目标 2 倍持续 15 分钟 | 停止扩量，限流/回滚。 |
| AI 失败率 ≥30% 持续 5 分钟 | 关闭 AI 能力或熔断，不下线手动经营。 |
| Worker 重复写/注销错误/导出跨空间 | 停止相关 Worker，保全审计，修复并回归。 |
| 备份/恢复门禁失败 | 阻断发布；已上线版本暂停后续变更。 |
