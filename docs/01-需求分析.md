## 1 引言

### 1.1 编写目的

本文档是 OPC AI OS（OPC AI 生态操作系统）项目**软件工程细化链路的第一份正式产物**。其目标：

1. 将《OPC AI OS 整体规划说明》中分散的战略表述，**收敛为可验证、可被技术架构继承的需求规格**；
2. 明确**全平台全景功能**与 **MVP（v1）范围**之间的边界，并对 v1 进一步拆分为**V1-A（核心价值验证版）**和**V1-B（商业闭环验证版）**两个子阶段；
3. 为后续技术架构设计、商业模式、发布路线图、合规文档提供**统一的需求基线**；
4. 通过"待决事项"清单，把尚未决定的策略性问题显性化，避免开发阶段返工；
5. **明确产品边界**——v1 服务谁、解决什么问题、不解决什么问题、最小成功条件是什么。

本文档**不包含**：技术选型、系统架构、Agent 编排框架、数据模型、API 设计、UI/UX 设计、商业定价方案。这些将分别由后续文档承担。

## 2 项目概述

### 2.1 项目定位

OPC AI OS 是面向**未来一人公司**的 **AI 原生商业基础设施**：

- **不是**单纯社区 SaaS（用户主动运营、关系自组织）；
- **不是**传统企业管理软件（用户主动操作工具完成流程）；
- **是**通过 AI 智能、商业协作、产业资源、数据资产，让 OPC 从"个人技能创业"走向"智能化经营"的操作系统。

平台采用 **"通用底座 + 行业插件 + AI 智能 + 数据资产"** 的架构模式。底层能力保持统一，不同类型的 OPC 社区可在其上接入专业插件。

### 2.2 产品愿景

> **让一个人拥有 AI 团队、商业网络、产业资源和数据智能，使一人公司的经营能力接近组织化企业。**

最终形态：**AI + OPC + 数据资产 + 产业生态** 构成的新型商业基础设施。

### 2.3 核心价值主张

| # | 价值 | 面向问题 |
|---|---|---|
| V1 | **AI 主动经营** | 从"用户操作软件"到"AI 主动推动经营"，OPC 无需具备全套企业经营技能 |
| V2 | **商业协作网络** | OPC 同时是需求方与供给方，平台撮合形成商业信用资产 |
| V3 | **行业插件生态** | 平台不直接覆盖所有行业，由行业伙伴提供专业能力 |
| V4 | **数据资产飞轮** | 经营数据持续沉淀，AI 越来越懂"OPC 如何成功经营" |
| V5 | **AI 合规服务** | 一人公司没有法务/合规岗，AI CLO 主动在关键节点为 OPC 把关 |

### 2.4 与现有方案的差异

| 维度 | 传统社区 SaaS | 经典企业管理软件 | 通用 AI Agent 平台 | **OPC AI OS** |
|---|---|---|---|---|
| 经营主动性 | 弱 | 无 | 中（被动响应） | **强（主动驱动）** |
| 商业网络 | 用户自组织 | 无 | 无 | **平台撮合** |
| 行业适配 | 单一垂直 | 单一垂直 | 通用 | **通用底座 + 行业插件** |
| 数据资产归属 | 平台 | 企业私有 | 用户自带 | **平台沉淀 + 用户授权** |
| 长期记忆 | 无/弱 | 企业数据库 | 弱 | **三层记忆（个人/共享/平台）** |

### 2.5 整体架构总览

按自上而下"用户感知 → 基础能力"排列：

```
┌─────────────────────────────────────────────┐
│ L1  OPC 用户生态层        （用户感知最近）  │
├─────────────────────────────────────────────┤
│ L2  AI Intelligence Layer（智能操作层）      │
├─────────────────────────────────────────────┤
│ L3  OPC 商业运营层  （任务/资源/服务三市场）│
├─────────────────────────────────────────────┤
│ L4  OPC 社区生态层                          │
├─────────────────────────────────────────────┤
│ L5  行业插件生态层                          │
├─────────────────────────────────────────────┤
│ L6  数据资产与 AI 底座层                    │
└─────────────────────────────────────────────┘
```

#### 2.5.1 架构演进版本对照

| 版本 | 架构描述 |
|---|---|
| 原始规划（REF-01） | 称"五层"，实际描述七层（含 Trust 层） |
| v0.1 | 统一为七层（含 L7 Trust 层） |
| v0.2 | 收敛为六层（取消 L7 Trust 层，功能归位至 L1/L3/L6） |
| v0.3 | 六层 + AI Memory & Data Flywheel 贯穿层 + M1 OPC 能力中台 |
| **v0.4** | **六层 + 贯穿层 + M1（内部平台服务层，不对外暴露为"插件平台"）；贯穿层定位调为"横向数据能力与治理机制"** |

#### 2.5.2 AI Intelligence Layer 定位（v0.4 修正）

AI Intelligence Layer 是平台的**统一智能入口之一**，负责自然语言理解、建议生成、任务编排和智能辅助。

用户**仍通过传统 UI 直接完成确定性业务操作**（如编辑任务、上传文件、确认验收）。所有高风险或不可逆操作必须经过明确授权和业务规则校验。

```
用户 → UI（直接操作） / AI（意图输入）
              ↓
       业务能力服务（L1–L6 各层）
              ↓
        数据与审计（贯穿所有层）
```

> **v0.4 修正说明**：v0.3 将 AI Intelligence Layer 描述为"用户与所有业务能力之间的唯一智能中枢"。此表述过于绝对，可能导致架构强耦合（所有业务操作必须经过 Agent，降低可控性和可测试性）。修正后 AI 是**入口之一**，而非唯一路径。

#### 2.5.3 AI Memory & Data Flywheel 定位（v0.4 修正）

AI Memory & Data Flywheel 是**横向数据能力与治理机制**，而非独立系统层：

```
行为数据 → 知识沉淀 → AI 记忆 → 智能反馈 → 更好的服务
        ↕        ↕        ↕        ↕        ↕
  ┌───────────────────────────────────────────────────────┐
  │              AI Memory & Data Flywheel                │
  │           （横向数据能力与治理机制）                    │
  └──────┬───────────┬──────────┬──────────┬──────────────┘
         ↓           ↓          ↓          ↓
   L1 用户生态  L2 AI Intel.  L3 商业运营  L4 社区生态
   L5 行业插件   L6 数据底座
```

> **v0.4 修正说明**：原"贯穿层"概念虽有战略表达价值，但易让研发团队误以为需要设计一个独立的系统层。修正为"横向数据能力与治理机制"，更贴近工程实现。其核心含义不变：每一层既是数据的生产者也是智能的消费者。

#### 2.5.4 架构概念归类（v0.4 新增）

| 分类 | 包含内容 |
|---|---|
| **业务域** | L1–L5（用户生态、AI 智能、商业运营、社区、行业插件） |
| **平台基础能力** | M1 能力中台（内部模块化服务，v3 前不对外开放） |
| **数据与智能基础设施** | L6（数据存储 + 模型推理 + 知识图谱 + AI Memory 中枢） |
| **横向治理能力** | Security / Compliance / Observability / Data Governance |

### 2.6 产品边界与核心假设（v0.4 新增）

#### 2.6.1 v1 服务对象

v1 仅服务于以下用户类型：

| 用户类型 | 是否 v1 目标 | 说明 |
|---|---|---|
| 个人创业者（1–2 人） | ✅ 主要目标 | 有核心技能，缺经营能力 |
| 微型团队（3–5 人） | ⚠️ 非优先 | v1 不支持企业级多成员协作，v2 起支持 |
| 企业用户 | ❌ 不支持 | v1 不提供企业实名、企业级权限、企业数据隔离 |
| 行业内已有成熟业务的团队 | ❌ 不支持 | v1 不绑行业，不提供行业专用能力 |

#### 2.6.2 v1 解决的问题

1. 个人创业者不知如何将想法拆解为可执行任务；
2. 个人创业者缺乏日常经营节奏，容易拖延和迷失优先级；
3. 个人创业者难以找到能力互补的合作方完成复杂项目；
4. 个人创业者缺乏客户管理意识，跟进散乱；
5. 个人创业者无法沉淀自己的技能、经验和商业信用。

#### 2.6.3 v1 不解决的问题（明确排除）

| 类别 | 排除原因 |
|---|---|
| 企业级多成员协作 | v1 工作空间为个人设计，不支持组织架构和企业成员管理 |
| 真实资金结算与自动抽佣 | 涉及支付牌照、资金监管、退款、争议处理，复杂度高 |
| 电子合同与在线签署 | 涉及法务有效性和存证要求，v2 起评估 |
| 自动信用分 | 需要足够交易数据积累后方可计算 |
| 行业插件生态 | 平台先验证核心价值，v3 起开放生态 |
| 多社区托管 | v1 仅平台自营单一社区 |
| AI 自动执行不可逆操作 | AI 仅提供建议，所有外部影响动作均需用户确认 |
| 复杂数据分析/财务预测 | 数据积累不足，AI 输出错误代价高 |
| 资源资产交易闭环 | 涉及权属、IP、侵权等复杂问题，v3 起评估 |

#### 2.6.4 v1 最小成功条件

V1-A 与 V1-B 分别设有阶段门（Stage Gate），达标方可进入下一阶段。

**统计规则（v0.4 修订）：**

各阶段分别定义观察条件，满足后方可判定通过：

**V1-A 通过门：**
- 不少于 **100 名有效激活用户**（完成注册+画像+创建项目）
- 连续观察至少 **2 周**
- 尚未达到 2 周的用户群数据标记为"数据未成熟"，不作为评审依据

**V1-B 通过门：**
- 满足以下**任一**样本条件（取先达到者）：
  - 不少于 **50 名有效需求发布用户**（至少发布 1 个需求）
  - 不少于 **30 笔完成首次合作**的记录
  - 不少于 **20 个有效合作样本**（即完成验收的合作）
- 观察窗口至少 **30 天**（因含"首次合作后 30 天内再次发起合作比例"指标）
- 尚未达到 30 天观察期的指标标记为"数据未成熟"

**通用规则：**
- 如样本量不足，不得直接判定通过；应标记为"数据不足"，延长观察周期或补充用户
- 指标拆分为三类：
  - **必须达标**（硬性条件，不达标不可通过）
  - **允许偏差**（可低于目标值但需说明原因）
  - **观察指标**（仅记录趋势，不作为通过条件）

| 阶段 | 类别 | 指标 | 目标 |
|---|---|---|---|
| **V1-A 通过门** | 必须达标 | 注册→首次创建项目转化率 | ≥50% |
| | 必须达标 | AI 任务建议确认率 | ≥30% |
| | 必须达标 | 创建项目后 7 日内再次使用率 | ≥35% |
| | 必须达标 | 每日三件事周活跃率 | ≥40% |
| | 必须达标 | AI 质量测试集通过率 | ≥90% |
| | 必须达标 | 重大安全事件数量 | 0 |
| | 必须达标 | 严重合规问题数量 | 0 |
| | 观察指标 | 任务完成率（确认后） | 记录趋势 |
| | 观察指标 | 用户主动修改 AI 任务比例 | 记录趋势 |
| | 观察指标 | AI 单用户成本 | 记录趋势 |
| **V1-B 通过门** | 必须达标 | 需求发布→合作确认转化率 | ≥30% |
| | 必须达标 | 首次合作后 30 天内再次发起合作比例 | ≥20% |
| | 必须达标 | 重大安全事件数量 | 0 |
| | 必须达标 | 严重合规问题数量 | 0 |
| | 允许偏差 | 撮合推荐命中率 | 记录趋势 |
| | 允许偏差 | 结算记录完成率 | 记录趋势 |

## 3 利益相关方与角色

### 3.1 平台角色体系


### 3.2 OPC 用户角色与画像

#### 3.2.1 初创 OPC（Persona #1，v1 主要目标用户）

| 维度 | 描述 |
|---|---|
| 典型描述 | 1–2 人微型创业单元，刚从雇佣转创业；有核心技能，缺经营能力 |
| 主要痛点 | 商业能力弱、缺客户、缺合作伙伴、不会拆任务、不会管项目 |
| 期望核心能力 | AI COO 把"想法/项目"拆解成可执行任务；每天获得"今天最重要的 3 件事" |
| 关键指标 | 任务拆解被采纳率、每日打开率、首单成交时间 |
| v1 是否付费 | 倾向免费基础功能 + AI 调用按量计费（详见商业模型文档） |

#### 3.2.2 成长型 OPC（Persona #2，v2 主要目标用户）

| 维度 | 描述 |
|---|---|
| 典型描述 | 3–5 人小组，已有 1–3 个稳定客户，开始产生协作需求 |
| 主要痛点 | 客户跟进混乱、项目交付不可控、合作方管理松散 |
| 期望核心能力 | 客户管理、项目管理、任务协作市场撮合 |
| 关键指标 | 协作撮合成交率、合作完成率、留存 |

#### 3.2.3 专业型 OPC（Persona #3，v3 主要目标用户）

已在 OPC 生态中具备一定声誉，承接较高客单价的服务，可能成为生态供给方。

#### 3.2.4 生态贡献者（远期）

成熟 OPC，主动分享经验、提供服务、培养新人。**不在 v1 范围**。

### 3.3 角色与权限矩阵（v0.4 新增）

以下矩阵定义 v1 各角色的数据访问范围和操作权限：

| 角色 | 数据访问范围 | 典型权限 | v1 是否存在 |
|---|---|---|---|
| **普通用户（OPC）** | 自己的数据 + 被授权的合作数据 | 创建项目、管理任务、发布需求、参与合作、上传文件、发布内容 | ✅ |
| **需求方** | 自己发布的需求 + 合作项目数据 | 选择候选方、确认验收、评价 | ✅（同一用户可同时为需求方和服务方） |
| **服务方** | 自己参与的合作数据 | 接受合作邀请、提交交付物、评价 | ✅ |
| **内容审核员** | 待审核内容队列 | 审核通过/驳回/标记，操作留痕 | ✅（轻量） |
| **客服** | 工单 + 有限用户信息（非敏感字段） | 查询、备注、协助处理，不可代用户操作 | ✅（轻量） |
| **平台管理员** | 平台配置 + 运营数据（不含用户私密数据） | 用户管理、平台配置、查看运营仪表盘、不可查看非授权用户私密数据 | ✅ |
| **合规/风控** | 风险事件 + 审核记录 + AI 审计日志 | 复核、处置、导出报告 | v2 |
| **插件管理员**（v3） | 自己插件的运行数据 | 插件上下架、版本管理 | v3 |

**权限设计原则（v1）：**

1. **最小权限**：用户默认只能访问自己的数据，合作数据需双方授权
2. **数据归属**：所有数据归属于创建者或其工作空间。用户保留其数据相关权益，平台仅在提供服务、履行法定义务或获得用户授权的必要范围内处理和使用数据，不因用户使用平台而当然取得超出必要范围的数据使用权
3. **管理员限制**：管理员可查看运营统计数据，但不可直接查看用户私密内容（私聊、非公开项目等）
4. **操作审计**：所有写操作均需鉴权，越权访问成功率目标为 0
5. **API 鉴权**：所有接口（除注册/登录外）均需鉴权

## 4 业务上下文

### 4.1 OPC 用户生命周期

```
S1 注册与定位
    ↓
S2 成长与赋能（AI CXO 持续陪伴）
    ↓
S3 交易与协作（撮合形成商业信用资产）
    ↓
S4 AI 经营阶段（AI 主动驱动日常经营）
    ↓
S5 生态贡献阶段（成熟 OPC 成为生态节点，远期）
```

**阶段与 MVP 的映射：**

| 阶段 | v1 支持 | 说明 |
|---|---|---|
| S1 注册与定位 | ✅ | 完整数字化画像 |
| S2 成长与赋能 | ⚠️ 部分 | 仅 AI COO；其余 CXO 不上 |
| S3 交易与协作 | ⚠️ 部分 | V1-A 仅项目管理，V1-B 开启任务撮合 |
| S4 AI 经营阶段 | ⚠️ 部分 | 仅"每日 3 件事 + 项目巡检" |
| S5 生态贡献阶段 | ❌ | 不在 v1 |

### 4.2 OPC 经营数据飞轮

```
用户行为（任意层：登录/创建项目/发布需求/上传内容）
    ↓
数据进入 AI Memory（L6 知识图谱 + RAG）
    ↓
AI 记忆被更新（用户画像 / 经营偏好 / 关系网络）
    ↓
智能反馈触发（更准的任务拆解 / 更匹配的推荐）
    ↓
用户感知到价值提升
    ↓
更多高质量行为（飞轮加速）
```

**三层记忆模型（v0.4 新增）：**

| 记忆层级 | 范围 | 默认策略 | 用户控制 |
|---|---|---|---|
| **个人记忆** | 仅服务当前用户或其工作空间 | 默认关闭 | 可查看/编辑/删除/关闭 |
| **共享记忆** | 经用户明确授权后用于团队或社区 | 默认关闭，需用户主动授权 | 可控制授权范围和期限 |
| **平台知识** | 脱敏、聚合、去标识化后用于公共能力 | 需在注册时单独授权 | 可随时撤回，撤回后停止将新的用户数据用于平台知识构建；已匿名化、聚合且无法与用户重新关联的数据不回溯删除 |

V1-A 不提供长期个人记忆；V1-B 上线长期记忆时默认关闭，用户主动启用后生效。

**数据授权规则：**

| 数据类型 | 默认是否进入个人记忆 | 是否允许平台学习 |
|---|---|---|
| 用户主动填写的画像 | 是 | 需单独授权 |
| 项目/任务 | 是 | 默认仅个人使用 |
| 客户信息 | 需用户确认 | 默认不用于平台训练 |
| 私聊内容 | 否 | 需明确授权 |
| 合同内容 | 默认仅业务隔离 | 不得默认用于训练 |
| 公开文章 | 可进入公共知识库 | 需版权和授权判断 |
| 交易评价 | 可用于信用 | 需告知用途范围 |

### 4.3 核心业务流程概览

#### 4.3.1 V1-A 核心流程：AI 驱动的个人经营

> **推送说明（v0.4 修正）**：V1-A 中"每日三件事"通过进入首页/工作台时展示，不实现站内信、短信、邮件及浏览器推送。站内通知中心和浏览器推送为 V1-B 能力。

```
注册 → 填写基础画像
    ↓
创建项目（输入几个字段）
    ↓
AI COO 自动拆解为多级任务
    ↓
用户确认/编辑任务列表
    ↓
进入工作台 → 查看今日最重要的 3 件事
    ↓
完成任务 → 项目进度自动更新
    ↓
AI COO 基于用户当前项目、任务、截止时间和客户记录提供下一步建议
```

#### 4.3.2 V1-B 核心流程：任务协作市场（如 V1-A 达标后启动）

```
需求方发布需求
    ↓
AI COO 协助拆解需求为子任务
    ↓
平台匹配合适的 OPC 候选（人工辅助推荐）
    ↓
需求方与候选方站内沟通
    ↓
合作确认（系统生成合作协议模板 + AI CLO 风险提示）
    ↓
执行交付（里程碑 + 交付物上传）
    ↓
验收 → 结算记录 → 双向评价
```

> 结算模式（v1-B）：系统仅记录合同金额和结算状态，不处理真实资金流转。详见 §6.3。

### 5.1 第一层：OPC 用户生态层（L1）

**层目标**：建立用户在平台的 OPC 数字身份、成长轨迹与商业关系网。

#### FR-L1-01 OPC 数字身份体系

**输入**：用户注册信息、能力自评、平台历史行为数据
**输出**：OPC 数字身份（个人/能力/创业阶段/商业目标）

### 5.2 第二层：AI Intelligence Layer（L2）

**层定位**：AI Intelligence Layer 是平台的统一智能入口之一，负责自然语言理解、建议生成、任务编排和智能辅助。

> v0.4 重申：AI 不是用户与业务能力之间的"唯一"中枢。用户仍通过传统 UI 直接完成确定性业务操作。

**层目标**：AI 主动辅助 OPC 经营。AI 角色（CEO/COO/CFO/CMO/CLO）是 L2 内部的 Agent 角色实例。

**重要决策**：v1 仅落地 AI COO（V1-A 即上线）与 AI CLO（V1-B 最小 2 触点）。AI CEO/CFO/CMO 不在 v1/v2 上线。

#### FR-L2-01 AI Agent 编排底座

> 支撑 AI COO 运行所需的最简 Agent 编排能力技术选型见技术架构文档。

| 功能项 | 优先级 | 验收要点 |
|---|---|---|
| 用户级 Agent 隔离 | 【v1-A】 | 不同用户的 Agent 状态互不污染 |
| Agent 短期记忆（当前会话） | 【v1-A】 | 单次对话内可引用前文 |
| Agent 长期记忆（简化版） | 【v1-B】 | 至少记住用户最近 30 天的关键事件 |
| Agent 运行日志可追溯 | 【v1-A】 | 每次 Agent 输出可追溯输入与依据 |
| Agent 远程调用第三方能力 | 【v2】 | 规范化插件协议，至少 1 个示例 |
| 多 Agent 协作 | 【v3】 | 协作场景文档化并可通过 |

#### FR-L2-02 AI COO（v1 主力 CXO 角色）

**职责**：日常运营管理——项目管理、任务拆解、工作安排、风险提醒。

> **数据读取边界（v0.4 新增）**：AI COO 通过受控业务查询可访问用户当前的项目、任务和客户数据，生成每日建议。V1-A 支持的三种数据类型：
> - ✅ **业务数据库查询** — 读取项目、任务、客户记录等结构化业务数据
> - ✅ **会话短期记忆** — 单次对话内引用前文
> - ❌ **AI 长期语义记忆** — 跨会话自动总结、历史行为自动记忆（V1-B）
> - ❌ **RAG 向量检索** — 基于知识库的检索增强生成（V1-B）
>
> "业务数据库查询"不属于"Agent 长期记忆"，两者在 V1-A 中严格区分。

| 功能项 | 优先级 | 验收要点 |
|---|---|---|
| 输入项目管理：根据用户输入，建立项目并拆解为多级任务 | 【v1-A】 | 见 §8.4 AI 质量评估标准 |
| 每日"今天最重要的 3 件事"在工作台首页展示 | 【v1-A】 | 用户进入首页/工作台时展示；不实现站内信/短信/邮件/浏览器推送；见 §6.7 指标定义 |
| 任务与项目状态联动 | 【v1-A】 | 任务完成可反映在项目进度 |
| 风险提醒（任务延期/客户逾期未跟进） | 【v1-B】 | 至少 2 类风险规则上线 |
| 自然语言驱动任务管理 | 【v2】 | 意图分类准确率 ≥90% |
| 主动性配置（用户可关闭/增强主动性） | 【v2】 | 状态可持久化 |
| AI 建议与自动执行的区分 | 【v1-A】 | AI 所有建议需用户确认后方可改变业务状态 |

#### FR-L2-03 AI CLO（合规顾问，v1-B 最小形态上线）

**职责**：在关键业务节点**主动触发合规与版权风险预检**。

**法律边界与责任说明（v0.4 修正）**：
- AI CLO 仅提供自动化风险提示和信息辅助，**不构成律师、法律顾问或专业合规机构出具的法律意见**
- 平台不保证 AI 提示的完整性、准确性或适用于所有具体场景
- 对一般风险提示，用户确认已阅读后可继续操作
- 对平台定义的高风险场景（涉及重大权益的合同条款等），应触发人工提醒或要求补充确认，具体规则由合规文档确定
- 本声明不排除平台依法应承担的责任；最终责任认定依据实际服务内容、是否存在误导性承诺、是否存在过错或过失、相关法律法规及司法管辖确定
- 每次 AI CLO 输出附带明确责任说明

| 功能项 | 优先级 | 验收要点 |
|---|---|---|
| 内容发布前合规预检（广告法/虚假宣传高频词） | 【v1-B】 | 用户点"发布"时调用，5 秒内返回风险评级与提示；可被忽略但留痕 |
| 合作协议模板填充时关键条款风险提示 | 【v1-B】 | 生成模板时附加"第 X 条需关注"标注与理由 |
| 免责声明展示 | 【v1-B】 | 每次 AI CLO 输出附带责任说明；一般风险用户确认后可继续，高风险场景（由合规文档定义）触发人工提醒或禁止继续操作 |

- CMO：增长功能强依赖流量入口与外部渠道，用户量未达阈值前 ROI 不可观

### 6.5 核心业务对象与状态机（v0.4 修订）

> **v0.4 拆分说明**：V1-A 个人任务与 V1-B 协作任务分属不同业务流程，状态机分离。同时新增项目、内容、合作、评价状态机。

#### 6.5.1 V1-A 个人任务状态（AI COO 生成，用户独立执行）

```
草稿 ──→ 待确认 ──→ 已确认 ──→ 进行中 ──→ 已完成 ──→ 已关闭
  │         │          │
  ▼         ▼          ▼
已取消    被驳回      已延期（标记）
```

| 状态 | 说明 | 允许操作 | 阶段 |
|---|---|---|---|
| 草稿 | AI 生成但未提交给用户 | 编辑、删除 | V1-A |
| 待确认 | 已展示给用户等待确认 | 确认、编辑、驳回 | V1-A |
| 已确认 | 用户确认可执行 | 开始执行 | V1-A |
| 进行中 | 正在执行 | 标记完成、标记延期 | V1-A |
| 已完成 | 用户自行标记完成 | 关闭 | V1-A |
| 已关闭 | 终结态 | — | V1-A |
| 已取消 | 用户主动取消 | — | V1-A |
| 被驳回 | AI 生成的任务被用户拒绝（区别于 V1-B 交付物驳回） | 编辑后重新提交 | V1-A |

**附加标记字段（非主状态）：**
- `是否延期`：超过原定截止时间
- `是否逾期`：超过宽限期仍未完成
- `用户是否拒绝`：用户明确拒绝执行

> V1-A 个人任务**没有**"待验收"和"争议中"状态——这些是 V1-B 协作任务的特性。

#### 6.5.2 V1-B 协作交付任务状态

```
待执行 ──→ 执行中 ──→ 待提交 ──→ 待验收 ──→ 已验收 ──→ 已关闭
  │                    │          │            │
  ▼                    ▼          ▼            ▼
已取消               已驳回     已驳回       争议中
```

| 状态 | 说明 | 允许操作 |
|---|---|---|
| 待执行 | 合作已确认，任务待开始 | 开始执行 |
| 执行中 | 服务方正在执行 | 提交交付物 |
| 待提交 | 已生成交付物待上传 | 上传交付物 |
| 待验收 | 已提交等待需求方验收 | 验收通过、驳回 |
| 已验收 | 需求方确认通过 | 关闭 |
| 已关闭 | 终结态 | — |
| 已取消 | 任一方主动取消（计入信用记录） | — |
| 已驳回 | 交付物不达标被退回 | 修改后重新提交 |
| 争议中 | 双方对完成质量有异议 | 提交争议说明，转人工客服 |

#### 6.5.3 项目状态

```
草稿 ──→ 进行中 ──→ 已完成 ──→ 已归档
  │         │
  ▼         ▼
已取消    已暂停
```

| 状态 | 说明 | 阶段 |
|---|---|---|
| 草稿 | 项目刚创建，尚未拆解任务 | V1-A |
| 进行中 | 已有任务在执行 | V1-A |
| 已完成 | 所有任务已完成 | V1-A |
| 已归档 | 项目关闭可归档 | V1-B |
| 已取消 | 用户主动取消 | V1-A |
| 已暂停 | 用户主动暂停 | V1-A |

#### 6.5.4 需求状态（V1-B）

```
草稿 ──→ 已发布 ──→ 匹配中 ──→ 沟通中 ──→ 已确定合作方 ──→ 执行中 ──→ 已完成 ──→ 已关闭
  │                     │                                    │
  ▼                     ▼                                    ▼
已撤回               匹配失败                              争议中
```

#### 6.5.5 结算状态（V1-B，人工辅助模式）

```
待结算（验收通过后） ──→ 已结算（线下完成，系统记录）
       │
       ▼
    争议记录 ──→ 转人工客服（V1-B 仅记录状态，不提供仲裁/赔付/法律判断）
```

> v2 接入支付后，结算状态机需重新设计为包含"待支付→已支付→冻结→已释放→退款中"等状态。

#### 6.5.6 合作状态（V1-B）

```
待确认 ──→ 已确认 ──→ 执行中 ──→ 已完成 ──→ 已关闭
  │         │          │
  ▼         ▼          ▼
已取消    已终止      争议中
```

#### 6.5.7 内容状态（V1-A 可选）

```
草稿 ──→ 待审核 ──→ 已发布 ──→ 已下架
              │
              ▼
           已驳回
```

#### 6.5.8 评价状态（V1-B）

```
待评价 ──→ 已评价 ──→ 已修改
                   已申诉
```

#### 6.5.9 AI 执行状态

```
已接收（用户触发）
  ↓
解析中 ──→ 等待用户补充（信息不足时）
  ↓
已生成建议（AI 完成分析，输出结果）
  ↓
等待用户确认 ──→ 已确认 ──→ 执行成功
       │                     │
       ▼                     ▼
     已拒绝                 执行失败（需降级处理）
```

**关键原则**：AI 建议 ≠ 业务状态变化。业务状态变化必须经过用户确认或业务规则校验。

### 8.2 AI 失败处理

| 场景 | 处理方式 |
|---|---|
| 模型超时（>15s） | 返回"正在处理中，请稍后查看"，后台异步重试 1 次 |
| 模型返回无效 JSON | 重试 1 次，仍失败则降级为自然语言输出并标记用户"请手动整理" |
| RAG 检索无结果（通用知识问答） | 明确告知用户"未找到相关数据"，可提供通用参考建议但必须标注"通用建议，非基于用户数据" |

| 第三方 API 不可用 | 跳过该能力，不影响整体服务 |
| 模型输出含明显错误 | 用户可"反馈错误"，反馈记录进入 AI 质量评估集 |

### 8.6 AI 决策审计链路（v0.4 新增）

AI 从触发到执行的全链路审计闭环：

```
用户触发 ──→ AI 接收（生成 traceId）
                ↓
           AI 处理（记录输入、模型版本、推理参数）
                ↓
           AI 输出建议（记录输出内容、引用来源）
                ↓
           等待用户确认（记录确认/修改/拒绝操作及时间）
                ↓
           ┌── 已确认 ──→ 执行（记录执行结果）
           │
           └── 已拒绝 ──→ 记录拒绝原因（如有）
```

| 环节 | 审计记录内容 | 存储 | 关联跟踪 |
|------|-------------|------|---------|
| 触发 | traceId、用户ID、触发时间 | 审计日志（append-only） | FR-CROSS-02 |
| 处理 | 输入摘要、模型版本、token 消耗 | AI 调用日志 | NFR-SEC-30 |
| 建议 | 输出内容摘要、引用来源（如有） | AI 调用日志 | FR-CROSS-02 |
| 确认 | 确认/修改/拒绝、操作时间、操作人 | 审计日志 | FR-CROSS-02 |
| 执行 | 执行结果、变更的业务状态 | 业务日志 | FR-CROSS-01 |
| 失败降级 | 失败原因、降级策略、用户可见提示 | 错误日志 | FR-CROSS-01 |

> 每条审计记录以 traceId 贯通，可在运营后台按用户/时间/操作类型筛选（V1-B 提供查询界面，V1-A 提供存储但不提供可视化界面）。

### 9.2 数据生命周期（v0.4 新增）

```
采集 → 使用 → 共享 → 存储 → 归档 → 删除
```

| 阶段 | 规则 |
|---|---|
| **采集** | 最小必要原则，仅采集产品功能必需的数据；采集前告知用途并获得用户同意 |
| **使用** | 数据仅用于用户授权的范围；平台内部使用需脱敏；外部使用需单独授权 |
| **共享** | 与合作方共享数据需双方确认；不经用户同意不向第三方提供（法定要求除外） |
| **存储** | 核心数据每日备份，保留 30 天；应用日志保留 180 天；审计日志保留 13 个月；AI 脱敏调用摘要保留 90 天 |
| **归档** | 账户注销后数据保留 30 天可恢复，到期后匿名化处理 |
| **删除** | 用户可申请数据导出和注销；法定要求留存的除外（如交易记录） |

| 数据类型 | 默认策略 | 说明 |
|---|---|---|
| 用户画像 | 账户存续期 | 注销后 30 天删除 |
| 实名认证结果 | 按实名服务协议和适用法规确定 | 仅存结果，不存原始证件 |
| 项目/任务 | 账户存续期 | 注销后 30 天删除 |
| 内容数据（动态/文章） | 账户存续期 | 用户可手动删除；注销后 30 天删除；审核记录按审计日志留存 |
| 评价数据 | 账户存续期 | 注销后对交易对方保留可见性但不显示实名；不可单独删除 |
| 合作记录 | 按业务和合同要求确定，法务确认后固化 | 涉及法律义务的从其规定 |
| 交易记录 | 按适用法律、财务和税务要求确定 | 法务确认后固化 |
| AI 调用记录 | 默认 90 天 | 必要时脱敏后延长保存 |
| 审计日志 | 默认 13 个月 | 安全事件、管理员操作和合规审计覆盖完整年度；法务要求更长期限时从其规定 |
| 私聊内容 | 账户存续期 | 用户可手动删除 |

### 9.3 数据授权与隐私

#### 9.3.1 数据使用权原则

用户保留其数据相关权益。平台仅在提供服务、履行法定义务或获得用户授权的必要范围内处理和使用数据，不因用户使用平台而当然取得超出必要范围的数据使用权。

| 原则 | 说明 |
|---|---|
| 必要范围 | 提供注册、登录、实名、项目管理、合作撮合等核心功能所必需的数据处理 |
| 授权范围 | 超出必要范围的数据处理（如模型改进、个性化推荐）须逐项获得用户单独授权 |
| 撤回权 | 用户可随时撤回非必要授权，撤回不影响已处理数据的合法性 |
| 透明性 | 用户可在设置中查看所有授权记录及范围 |

#### 9.3.2 授权类型

详见 FR-L6-04 授权类型说明表。各授权类型相互独立，不得使用单一总勾选框替代。

#### 9.3.3 记忆相关授权

详见 §4.2 三层记忆模型（个人记忆/共享记忆/平台知识）。

### 9.4 数据导出与注销

| 功能 | v1 实现 |
|---|---|
| 基础数据导出 | CSV 格式，包含项目、任务、客户数据 |
| 完整数据导出（v2） | 一键导出全部数据 |
| 账户注销申请 | 提交后停止新数据处理，30 天宽限期（可恢复） |
| 法定留存数据 | 因法律、争议或安全原因需继续保留的数据，向用户说明原因和期限；其余数据匿名化 |
| 已脱敏聚合数据 | 已匿名化、聚合且无法与用户重新关联的数据，不回溯删除 |
| AI 记忆 | 注销后同步删除或匿名化（无法与用户关联的已脱敏数据除外） |
| 合作对方可见性 | 注销后已完成合作记录对交易对方仍可见（但不显示已注销用户的实名信息） |
| 备份中的数据处理 | 备份中的数据在下次备份周期覆盖时清除，最长不超过备份保留期 |
| 第三方服务数据删除 | 平台代用户向对接的第三方发送删除请求（v2） |

## 10 约束与假设

### 10.1 约束

| C-01 | 合规 | 用户与经营数据存储须满足中国数据安全法与个保法；AI CLO 提示不构成法律意见，但不排除平台依法应承担的责任 |
|---|---|---|
| C-02 | AI 责任 | AI 仅生成建议，所有影响外部世界的动作须经用户确认 |
| C-03 | 第三方依赖 | 实名认证依赖第三方可用性；v1 仅接实名认证 |
| C-04 | 团队规模约束 | v1 团队预计较小（≤10 人 FTE） |
| C-05 | 资金约束 | v1 不引入重基础设施投入（自训大模型/私有化部署不进 v1） |

### 10.2 假设

| A-01 | AI 模型可用性 | 商业 LLM API 在 1 年内可用性 ≥99% |
|---|---|---|
| A-02 | 用户对 AI 信任 | 假设 OPC 用户愿意接受 AI 任务拆解并执行 |
| A-03 | 撮合供给侧 | v1-B 需积累足够供给侧用户；冷启动由产品/运营手动撮合标杆案例 |
| A-04 | OPC 付费意愿 | 假设 OPC 在享受价值后愿意付费 |
| A-05 | V1-A→V1-B 阶段门 | V1-A 指标达标是进入 V1-B 的前提条件；如 V1-A 核心假设不成立，V1-B 不启动 |
